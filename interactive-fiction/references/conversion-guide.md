# 线性→互动转化实战指南

> 怎么把一部写好的悬疑小说拆成节点树
> 包含：章节分解法、高潮多路径适配、范围估算

---

## 一、章节分解启发式

### 一章变几个节点？

```
线性章节类型 → 节点数量：

调查/审讯章节（2500字）→ 3-5 个节点
  开头描写（1 个叙事节点）
  + 核心互动（1-2 个决策节点）
  + 结果分支（1-2 个叙事节点）

纯叙事/氛围章节（2000字）→ 1-2 个节点
  大部分保留为一个长叙事节点
  + 可选：末尾一个轻决策点

高张力/追逐章节（1500字）→ 2-4 个节点
  短节点快节奏（200-400字/节点）
  + 多个紧急决策点

揭示/高潮章节（3000字）→ 2-3 个节点
  保留为较长的叙事节点（质量优先于互动量）
  + 关键的指控/选择（1 个高风险决策点）
```

### 在哪里切割？

```
一章文本中，寻找这些"天然切割点"：

1. 角色做出选择的地方
   "他决定去找陈医生。" → 改为：让读者决定去找谁

2. 到达新地点的地方
   "她推开仓库的门。" → 新节点开始

3. 发现重要信息的地方
   "信封里是一张照片。" → 让读者选择先看什么

4. 对话中的关键转折
   "你到底知不知道她的秘密？" → 对话选择点

5. 时间跳跃处
   "三小时后——" → 天然的节点边界

不要在这些地方切割：
  ✗ 一段描写的中间（破坏氛围连续性）
  ✗ 一段对话的中间（除非是决策点）
  ✗ 角色内心独白的中间（破坏思路）
```

## 二、高潮多路径适配

### 核心问题

```
线性小说：一个精心设计的揭示场景
互动小说：读者走了不同路线，掌握不同信息

→ 同一个揭示在不同信息状态下必须都"有效"
```

### 三种适配策略

**策略 A：共享揭示 + 条件细节（推荐）**

```
所有路线汇聚到同一个揭示场景。
核心真相相同。
但揭示的"方式"和"细节"根据路线变化。

示例：
  所有路线："凶手是陈医生"

  路线1（仓库线索 + 高信任）：
    陈医生在团队面前主动坦白。
    揭示方式：自白 + 侦探的理解

  路线2（证人线索 + 低信任）：
    侦探用证据逼迫陈医生认罪。
    揭示方式：对质 + 证据展示

  路线3（混合线索 + 中信任）：
    侦探推理展示 + 陈医生沉默确认。
    揭示方式：推理 + 沉默
```

**策略 B：多结局揭示（复杂但高回报）**

```
不同信息量 → 不同的揭示深度 → 不同的结局

  信息充足（证据 ≥ 8）：完整揭示 + 正义实现
  信息中等（证据 4-7）：部分揭示 + 遗留谜团
  信息不足（证据 < 4）：错误指控 / 无法破案

每个结局都是"完整的故事"——不是 game over。
```

**策略 C：渐进揭示（高潮拆分）**

```
不把揭示放在一个节点里。
拆成 3-5 个揭示节点，读者通过选择"拼出"真相。

  揭示节点1：动机揭示（为什么）
  揭示节点2：手法揭示（怎么做到的）
  揭示节点3：身份揭示（是谁）

  读者选择揭示的顺序。
  不同顺序产生不同的情感体验。
  但最终都到达完整真相。
```

## 三、范围估算

### 从线性到互动的工作量公式

```
互动版总字数 ≈ 线性版字数 × 分支系数

分支系数（根据拓扑模型）：
  瓶颈型：1.5-2.0x
  平行调查型：2.0-3.0x
  信任网络型：2.5-4.0x

示例：
  线性小说 3万字 × 瓶颈型 1.8x = 互动版约 5.4万字
  → 约 60-80 个节点
  → 8-12 个决策点
  → 3-4 个结局
```

### 时间估算

```
每个节点的工作量：

骨架设计（frontmatter + 场景描述）：5-10 分钟
正文写作（800-2000字）：30-60 分钟
条件文本（汇聚节点额外）：15-30 分钟
验证（与前后节点一致性检查）：10-15 分钟

→ 平均每节点：1-2 小时

60 节点项目 ≈ 60-120 小时工作量
（AI 辅助可缩短至 30-60 小时）
```

### 何时缩减范围

```
缩减信号：
  □ 决策点超过 15 个 → 考虑合并相似的选择
  □ 结局超过 6 个 → 考虑合并差异不大的结局
  □ 同一信息在 3+ 条路线中重复 → 提取到汇聚节点
  □ 某条分支的独有内容 < 500字 → 用条件文本替代独立分支

缩减技术：
  1. 将完整分支降级为条件文本（在汇聚节点中用 if/else）
  2. 将 3 选项简化为 2 选项
  3. 合并效果相似的选项（不同台词但同一后续节点）
  4. 用状态变量记录选择，但不创建独立路线
```

## 四、悬疑线索在分支中的分配

### 线索分配规则

```
关键线索（推理真相必需的）：
  □ 每条关键线索至少在 2 条路线上可获得
  □ 不同路线可以用不同方式获得同一线索
  例：路线A在仓库直接发现凶器
      路线B从证人口述中推断凶器存在

辅助线索（增强理解但非必需的）：
  □ 可以只在 1 条路线上出现
  □ 作为"该路线的独特奖励"增加重玩价值

红鲱鱼：
  □ 至少在 1 条路线上被消解
  □ 另一条路线可以不碰到这个红鲱鱼（也是一种"消解"）
```

### 线索-路线矩阵

```
              路线A  路线B  路线C
  线索C01     ✓直接  ✓间接   ✗
  线索C02     ✗      ✓直接  ✓直接
  线索C03     ✓直接  ✗      ✓间接
  线索C04     ✓直接  ✓直接  ✓直接  ← 关键线索，所有路线可得

  验证：每条到达揭示的路线，是否拥有足够线索推理出真相？
  路线A: C01 + C03 + C04 = 可推理 ✓
  路线B: C01 + C02 + C04 = 可推理 ✓
  路线C: C02 + C03 + C04 = 可推理 ✓
```

## 五、完整转化工作清单

```
Phase 1: 分析原作（1-2天）
  □ 标记所有角色决策点
  □ 标记所有调查行动（可选方向）
  □ 标记所有"读者会想'我会怎么做'"的时刻
  □ 绘制线索分布图
  □ 确定转化策略（A/B/C）和拓扑模型
  □ 范围估算：节点数、字数、工时

Phase 2: 架构设计（1-2天）
  □ manifest.yaml：所有节点 ID + 依赖关系
  □ variables.yaml：状态变量注册表
  □ 线索-路线矩阵
  □ 结局触发条件表
  □ Mermaid 流程图

Phase 3: 骨架填充（每节点 10 分钟）
  □ 所有节点的 frontmatter + 场景描述 + 选择文案
  □ 不写正文——只确保结构完整

Phase 4: 正文写作（每节点 30-60 分钟）
  □ 按章节顺序逐节点写作
  □ 使用上下文注入协议（story-bible + codex + rolling summary）
  □ 每写完一章更新摘要和 codex

Phase 5: 验证（1-2天）
  □ 结构验证（frontmatter 一致性）
  □ 5 条关键路径通读
  □ 线索公平性验证
  □ 导出并实际运行测试
```
