# 动态互动叙事

> 超越静态分支：LLM 驱动的实时叙事生成
> 基于 Dramamancer (UIST 2025)、Drama Llama (2025)、WhatELSE (CHI 2025)、StoryVerse (FDG 2024)、Drama Machine (2024)

---

## 一、从静态分支到动态生成的范式转移

```
静态分支（我们之前的模式）：
  作者写好所有节点 → 玩家在预定选项中选择 → 走预定路径
  优势：质量可控、叙事精确
  劣势：内容量爆炸、玩家只能在预设选项中选择、无法处理意外输入

动态生成（新范式）：
  作者定义约束/意图/模式 → LLM 在运行时根据玩家行为实时生成叙事
  优势：无限可能性、真正的玩家自由、内容量可控
  劣势：质量难以保证、可能偏离作者意图、一致性挑战

最佳实践：混合模式
  关键剧情节点 = 静态（作者精确控制）
  节点之间的过渡 = 动态（LLM 生成）
  玩家输入 = 自由文本（LLM 理解并整合）
```

## 二、Story Schema 模式（Dramamancer）

### 核心架构

```
作者不写"节点"，而是写"Schema"：

Story Schema = {
  style: "叙事风格指令",
  characters: [角色描述列表],
  scenes: [
    {
      setting: "场景设定",
      characters: [本场景角色],
      opening: "开场文本",
      events: [
        {
          condition: "自然语言条件（当什么情况发生时）",
          outcome: "自然语言结果（触发什么叙事后果）"
        }
      ]
    }
  ]
}
```

### 双模块运行时

```
每一轮互动：

1. 玩家输入自由文本（行动/对话/任何事）
     ↓
2. Instantiation Module（实例化模块）
   输入：故事历史 + Schema约束 + 玩家输入
   输出：下一段叙事文本
   规则：必须回应玩家输入（不能忽略），同时尊重风格/角色/场景约束
     ↓
3. Interpretation Module（解释模块）
   输入：当前叙事状态 + 所有未触发的 event conditions
   输出：哪些 event conditions 现在为真？
   → 被触发的 event outcomes 注入下一轮的生成上下文
     ↓
4. 循环回到 1
```

### 悬疑适配

```
悬疑 Story Schema 示例：

{
  style: "冷硬写实，第三人称有限视角，短句为主",
  characters: [
    { name: "林探长", role: "player", desc: "疲惫但执着的老刑警" },
    { name: "陈医生", role: "npc", desc: "受害者的主治医生，隐藏着什么" },
    { name: "周助理", role: "npc", desc: "新来的法医助理，过于热心" }
  ],
  scenes: [
    {
      setting: "医院停尸房，凌晨2点",
      characters: ["林探长", "周助理"],
      opening: "停尸房的灯管闪了两下。冷气从脚踝开始往上爬。",
      events: [
        {
          id: "E01_discover_injection",
          condition: "玩家检查了受害者的颈部或手臂",
          outcome: "发现一个极小的注射痕迹——验尸报告中没有提到这个。"
        },
        {
          id: "E02_zhou_nervous",
          condition: "玩家询问周助理关于验尸报告的问题",
          outcome: "周助理的解释流畅得不自然——像排练过的。"
        },
        {
          id: "E03_chen_arrives",
          condition: "场景持续超过5轮对话 或 玩家试图离开",
          outcome: "门开了。陈医生站在门口：'这么晚了，林探长还在？'"
        }
      ]
    }
  ]
}

关键：events 是"漂浮的"——它们不按顺序，
而是在条件满足时随时触发。
玩家可能先触发 E03 再触发 E01，或永远不触发 E02。
```

## 三、NL-Trigger Drama Manager（Drama Llama）

### 自然语言条件替代布尔逻辑

```
传统 storylet：
  if trust_chen > 50 AND found_knife == true AND location == "office":
    → 触发"陈某主动告白"storylet

Drama Llama 方式：
  trigger: "当玩家已经与陈某建立了一定信任，
           并且玩家已经发现了凶器（或暗示知道凶器的事），
           并且场景在一个相对私密的环境中"
  stage_direction: "陈某犹豫了一下，然后说出了ta一直隐瞒的事..."

LLM 评估这个自然语言条件是否在当前叙事上下文中成立。
不需要精确的变量——LLM 理解叙事语义。

优势：
  - 作者不需要编程
  - 条件可以是模糊的/主观的（"当气氛变得紧张时"）
  - 适应玩家的自由行为

劣势：
  - LLM 判断可能不一致
  - 需要良好的 prompt 工程
  - 复杂条件可能被误判
```

### 悬疑特化的 Drama Manager

```
悬疑 drama manager 需要额外管理：

1. 线索释放控制
   不能让 LLM 过早揭示真相
   → 添加"锁定条件"：某些信息在特定事件触发前绝对不能出现

2. 嫌疑方向控制
   LLM 生成的 NPC 对话可能意外暴露犯人
   → 添加"NPC 知识限制"：每个 NPC 只能说出 ta 知道的信息

3. 公平性保证
   动态生成的线索是否仍然"公平"？
   → 添加"线索注入器"：确保关键线索在一定时间内被呈现
   → 如果玩家 N 轮没有发现关键线索，drama manager 主动注入一个发现机会

4. 张力曲线维护
   LLM 自然倾向于过快解决冲突
   → 添加"张力监控"：当张力低于目标曲线时注入新障碍/威胁
```

## 四、抽象行为实例化（StoryVerse）

### Abstract Acts 模式

```
作者不写具体场景，而是写"抽象行为"：

Abstract Act: {
  goal: "侦探发现受害者有一个秘密身份",
  prerequisites: ["侦探已经审问过受害者的家人"],
  variables: {
    discovery_method: "未指定",  // LLM 自行决定
    emotional_reaction: "未指定"
  },
  constraints: [
    "发现必须通过间接方式（不能直接告知）",
    "侦探的反应必须反映 ta 的个人创伤"
  ]
}

LLM Act Director 将抽象行为实例化为具体叙事：
  → 可能实例化为"在受害者的书房发现了一本伪造的护照"
  → 也可能实例化为"一个陌生人来认领遗体，说出了一个不同的名字"
  → 取决于当前世界状态和玩家的之前行为

核心原理：作者控制"什么必须发生"，LLM 决定"怎么发生"。
```

### 在悬疑中的应用

```
悬疑 Abstract Acts 示例：

Act 1: {
  goal: "建立核心谜团",
  events: [
    { what: "发现案件", how: "未指定", when: "开场10分钟内" },
    { what: "引入3个嫌疑人", how: "未指定", when: "前30%内" },
    { what: "布设线索C01和C02", how: "未指定" }
  ]
}

Act 2: {
  goal: "调查并复杂化",
  events: [
    { what: "排除第一个嫌疑人", trigger: "玩家调查了嫌疑人A" },
    { what: "中点翻转", trigger: "玩家发现了线索C03 或 时间过半" },
    { what: "侦探面临个人危机", trigger: "中点翻转后" }
  ]
}

这些 Acts 是"骨架"——LLM 在骨架之间填充自由叙事。
```

## 五、多Agent角色模拟（Drama Machine）

### Ego/Superego 双Agent模型

```
每个NPC由两个Agent驱动：

Ego Agent（表面对话）：
  - 处理与玩家的直接交流
  - 维持角色的公开人设
  - 回应玩家的问题和行为

Superego Agent（内心独白）：
  - 运行角色的内心活动
  - 评估"我是否在暴露太多？"
  - 决定角色是否应该说谎/回避/转移话题

两者的交互：
  玩家提问 → Ego 生成初始回答
  → Superego 审查："这个回答会暴露我的秘密吗？"
  → 如果会 → Superego 指示 Ego 修改回答（回避/说谎/转移）
  → 最终输出到玩家

悬疑中的价值：
  犯人的 Ego 想表现得正常 → Superego 不断压制真相
  → 但在压力下 Superego 可能失灵 → 露出破绽
  → 这个"破绽"就是动态生成的线索！
```

## 六、自由输入处理

### 超越选项的玩家输入

```
静态模式：玩家从 2-3 个选项中选择
动态模式：玩家输入任何自然语言文本

处理架构：

玩家输入: "我想检查桌子下面"
  ↓
1. 意图识别：行动类（探索-检查-桌子下方）
2. 世界状态检查：桌子下面有什么？
   → 如果 Schema 中定义了：返回预定义内容
   → 如果没有定义：LLM 生成合理的发现（受约束）
3. 线索注入检查：是否有待释放的线索可以放在这里？
4. 戏剧性评估：这个发现现在是否有戏剧效果？
5. 生成叙事输出

悬疑特化的输入类型：
  - 调查行为（搜查、检查、取证）→ 可能触发线索
  - 对话行为（审问、套话、安慰）→ 可能触发信息释放
  - 推理行为（指控、假设、推论）→ 可能触发嫌疑确认/否定
  - 社交行为（建立信任、威胁、贿赂）→ 影响关系状态
  - 元行为（存档、查看笔记、回顾线索）→ 系统功能
```

## 七、混合模式设计（推荐架构）

### 静态+动态最佳组合

```
             ┌──── 动态区间 ────┐
静态节点 A → │ LLM 自由生成     │ → 静态节点 B → │ LLM 自由生成 │ → 静态节点 C
 (关键剧情)   │ 玩家自由输入     │   (关键剧情)    │ 玩家自由输入 │   (揭示)
             │ Drama Manager    │               │              │
             │ 监控触发条件     │               │              │
             └──────────────────┘               └──────────────┘

静态节点：作者完全控制的关键剧情点
  → 案件发现、中点翻转、关键线索、揭示场景
  → 质量有保证，叙事精确

动态区间：LLM + 玩家共同驱动
  → 调查过程、NPC 互动、环境探索
  → 自由度最大，体验独特

Drama Manager：在动态区间中维护作者意图
  → 监控是否偏离太远
  → 在适当时机推动向下一个静态节点过渡
  → 确保关键线索被释放

这就是 Drama Llama 的 "branch-and-bottleneck" 的动态版本。
```

### 悬疑混合模式的约束层级

```
Level 0：不可违反（硬约束）
  - 真凶身份不会改变
  - 关键线索必须在揭示前被释放
  - 角色不能说出超出自己知识状态的信息

Level 1：强烈偏好（软约束）
  - 嫌疑方向应该按照设计的曲线变化
  - 张力应该整体上升
  - 主题应该被持续体现

Level 2：建议（弱约束）
  - NPC 对话风格应该一致
  - 场景描写应该维持调性
  - 节奏应该符合当前戏剧阶段

Level 3：自由（无约束）
  - 具体措辞
  - 环境细节
  - NPC 的非关键反应
  - 场景过渡方式
```

## 八、评估维度（动态叙事特有）

### 作者视角评估

```
□ 风格一致性：生成的文本是否符合 Schema 的风格指令？
□ 角色辨识度：每个角色是否保持独特的声音？
□ 场景意识：生成内容是否尊重当前场景设定？
□ 事件检测准确率：条件满足时是否正确触发？
□ 事件实现质量：触发后的叙事是否自然融入？
□ 线索释放控制：是否过早或过晚释放关键信息？
```

### 玩家视角评估

```
□ 输入响应性：玩家的行为是否被有意义地回应？
□ 代入感：玩家是否感到"我在这个世界中"？
□ 选择重要性：玩家的决定是否产生了可感知的影响？
□ 推理参与感：玩家是否感到自己在"真的推理"？
□ 惊奇感：是否有玩家没有预料到的发展？
□ 公平感：揭示时玩家是否觉得"我本来能推理出来"？
```
