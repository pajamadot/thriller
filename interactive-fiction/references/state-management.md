# 互动叙事状态管理

> 跟踪读者选择、管理变量、触发条件逻辑
> 参考 Twine/ink/ChoiceScript 的状态系统设计

---

## 一、状态变量设计

### 变量类型

| 类型 | 用途 | 示例 | 适用场景 |
|------|------|------|---------|
| 布尔型 | 是/否标记 | `found_knife = true` | 线索发现、事件触发 |
| 整数型 | 计数/评分 | `evidence_count = 5` | 证据收集、好感度 |
| 范围型 | 有上下限的数值 | `trust_chen: 0-100` | 关系状态 |
| 枚举型 | 多选一状态 | `accusation: none/A/B/C` | 关键决策记录 |
| 集合型 | 已收集的物品/信息 | `clues: [c01, c03, c07]` | 线索收集系统 |

### 变量命名规范

```
前缀规范：
  found_   → 布尔，是否发现某物
  know_    → 布尔，是否知道某信息
  trust_   → 范围，对某角色的信任度
  rel_     → 范围，与某角色的关系
  sus_     → 范围，对某角色的怀疑度
  has_     → 布尔，是否持有某物
  did_     → 布尔，是否做过某事
  count_   → 整数，某类事物的数量
  flag_    → 布尔，流程控制标记

示例：
  found_murder_weapon     → 是否发现凶器
  know_victim_secret      → 是否知道受害者的秘密
  trust_detective_wang    → 对王警探的信任度
  did_search_basement     → 是否搜查了地下室
  count_evidence          → 收集的证据总数
```

### 变量数量控制

```
推荐变量数量：

  轻量互动（3-5决策点）：5-10 个变量
  标准互动（8-12决策点）：15-25 个变量
  深度互动（15+决策点）：30-50 个变量

过多变量的问题：
  - 状态组合爆炸，难以测试
  - 容易出现矛盾状态
  - 读者无法追踪自己的选择

过少变量的问题：
  - 选择感觉无意义
  - 路线差异不够
  - 结局分化不足
```

## 二、条件逻辑设计

### 条件类型

```
1. 门控条件（Gate Condition）
   进入某节点的前提条件
   例：if found_secret_room → 解锁"密室调查"节点

2. 变体条件（Variant Condition）
   同一节点内的文本差异
   例：if trust_chen > 60 → 陈某主动分享信息
       else → 陈某保持沉默

3. 结局条件（Ending Condition）
   触发特定结局
   例：if count_evidence >= 7 AND know_motive → 完美结局

4. 累积条件（Accumulation Condition）
   多个选择的累积效果
   例：if trust_a + trust_b + trust_c >= 150 → 团队合作路线
```

### 条件嵌套深度

```
推荐最大嵌套深度：2 层

  ✓ 好：if found_knife AND did_interview_chen
  ✓ 好：if trust_wang > 50 AND NOT did_betray_wang
  ✗ 过深：if (found_knife AND did_interview_chen) AND
              (trust_wang > 50 OR (know_secret AND did_search))

过深的条件 → 简化为中间变量：
  set investigation_complete = found_knife AND did_interview_chen
  set wang_alliance = trust_wang > 50 AND NOT did_betray_wang
  if investigation_complete AND wang_alliance → ...
```

## 三、状态转换设计

### 状态变更时机

| 时机 | 描述 | 示例 |
|------|------|------|
| 选择时 | 做出选择立即改变 | 选择信任 A → trust_a += 20 |
| 到达时 | 到达某节点时改变 | 进入密室 → found_secret_room = true |
| 条件满足时 | 达到某条件自动触发 | evidence >= 5 → flag_can_accuse = true |
| 对话中 | 对话选项改变状态 | 告诉 A 秘密 → know_a_knows_secret = true |

### 状态变更的叙事反馈

```
每次状态变更应有对应的叙事体现：

显性反馈（读者能感知）：
  "王警探的表情微妙地变化了。他似乎更信任你了。"
  → trust_wang += 15

隐性反馈（读者可能感知）：
  （下一场对话中，王警探主动分享了一条信息）
  → 不直接说明信任增加，但通过行为体现

零反馈（读者不感知）：
  flag_visited_room_3 = true
  → 纯流程控制，不需要叙事反馈
```

## 四、状态可见性策略

### 悬疑互动小说的特殊考虑

```
应该对读者可见的状态：
  - 已收集的线索/证据（推理需要）
  - 可以采取的行动（避免盲目尝试）
  - 时间/资源限制（建立紧迫感）

应该对读者隐藏的状态：
  - 角色信任度的精确数值（保持不确定性）
  - 嫌疑指向系统（避免元游戏）
  - 结局条件（保持悬念）
  - 死亡/危险阈值（维持恐惧感）

半透明状态（读者隐约感知）：
  - 角色态度变化（通过行为表现，不用数字）
  - 调查进度（通过叙述暗示，不用进度条）
  - 危险程度（通过氛围描写）
```

### 信息面板设计（如果平台支持）

```
推荐显示：
  📋 线索笔记本
     - 已发现的线索列表
     - 可点击查看详情

  👤 嫌疑人档案
     - 已了解的信息
     - 不在场证明状态

不推荐显示：
  ✗ 信任度百分比
  ✗ "距离结局还有 X 步"
  ✗ "最优路线提示"
```

## 五、结局触发系统

### 结局分级

```
S 级结局（完美/真相结局）：
  条件：最严格，需要大量关键状态满足
  体验：最完整的真相揭示，最好的情感回报
  可达性：~15% 的读者首次到达

A 级结局（好结局）：
  条件：中等严格
  体验：核心真相揭示，有遗留但令人满意
  可达性：~30% 的读者首次到达

B 级结局（中性结局）：
  条件：宽松
  体验：部分真相，部分遗憾
  可达性：~35% 的读者首次到达

C 级结局（坏结局）：
  条件：默认/最低
  体验：失败但有叙事价值
  可达性：~20% 的读者首次到达

设计原则：即使是 C 级结局也应该是"好的故事"，不是"游戏结束"
```

### 结局触发矩阵

```
| 结局 | 证据量 | 信任系统 | 关键事件 | 特殊条件 |
|------|--------|---------|---------|---------|
| S1 | ≥8 | trust_key≥70 | all_alive | found_secret |
| A1 | ≥6 | trust_key≥50 | - | - |
| A2 | ≥7 | - | saved_witness | - |
| B1 | ≥4 | - | - | - |
| B2 | 3-5 | low_trust | - | did_wrong_accuse |
| C1 | <3 | - | - | - |
| C2 | any | - | detective_caught | sus_level≥90 |
```

## 六、调试与测试

### 状态测试清单

```
1. 可达性测试
   □ 每个结局都可以通过合理的选择到达
   □ S 级结局不需要"完美攻略"才能到达
   □ 没有不可能满足的条件组合

2. 一致性测试
   □ 没有矛盾的状态组合
     例：found_body = false 但 called_police_about_body = true
   □ 角色的行为在所有状态组合下都自洽
   □ 条件检查使用的变量都已在之前的节点中设置

3. 边界测试
   □ 所有数值变量都不会超出预期范围
   □ 布尔变量的翻转逻辑正确
   □ 集合变量不会包含重复项

4. 路径测试
   □ 最短路径体验完整
   □ 最长路径不会过于冗长
   □ 随机选择路径 10 次，每次体验都自洽
```
