# 互动悬疑小说创作系统 — Interactive Thriller Fiction

> 将悬疑小说转化为带分支选择的互动叙事体验
> 融合 Choice Architecture（选择架构）、Narrative Design（叙事设计）、Game Writing（游戏写作）方法论

---

## 系统定位

本技能将线性悬疑小说转化为**多分支互动小说**，读者在关键时刻做出选择，影响调查方向、角色命运和最终结局。适用于：

- 互动小说平台（Twine / ink / ChoiceScript / 橙光 / 快点阅读）
- 互动影视脚本（Netflix 交互式 / 短剧互动版）
- 文字冒险游戏
- EPUB 互动增强版

---

## 核心理念

### 三大设计支柱

1. **有意义的选择**：每个分支选择必须让读者感受到"我的决定改变了什么"
2. **叙事完整性**：无论走哪条路线，故事都必须在逻辑和情感上完整
3. **可控复杂度**：分支数量在创作可行性范围内，避免指数爆炸

### 悬疑类型的特殊优势

悬疑是最适合互动化的文学类型：
- **调查天然是选择**：先查哪条线索？先审谁？
- **信息即权力**：不同路线获取不同信息，影响推理
- **多结局有天然锚点**：真凶已定，但读者能否找到 ta？
- **失败有叙事价值**：推理错误不是"game over"而是"另一个故事"

## 命令体系

### `/init` — 项目设置

**输入要求**：

| 参数 | 说明 | 选项 |
|------|------|------|
| 来源 | 素材来源 | 已有小说（提供文本）/ 从零创作 / 基于大纲 |
| 分支深度 | 复杂度级别 | 轻量(3-5个决策点) / 中等(8-12个) / 深度(15+个) |
| 结局数量 | 目标结局数 | 2-3个 / 4-6个 / 7+个 |
| 平台 | 目标发布平台 | Twine / ink / ChoiceScript / 橙光 / 纯文本 / 通用 |
| 互动模式 | 读者角色定位 | 侦探视角 / 上帝视角 / 旁观者（详见下方） |

**三种互动模式详解**：

| 模式 | 读者身份 | 选择类型 | 信息获取 | 适合 |
|------|---------|---------|---------|------|
| **侦探视角** | 读者=主角 | 调查方向、审讯策略、推理指控 | 只看到主角看到的 | 本格、犯罪程序、密室 |
| **上帝视角** | 读者=导演 | 控制多个角色的行动 | 看到所有角色的视角 | 社会派、多视角叙事 |
| **旁观者** | 读者=摄影机 | 选择跟随谁、关注什么 | 每次只看一个视角但可切换 | 心理悬疑、信任网络 |

上帝视角设计要点：
- 每个决策点标注"你在控制谁的行动"
- 信息在角色之间不自动共享（角色 A 发现的线索，角色 B 不知道）
- 选择的张力来自"知道全局但每个角色不知道"的戏剧反讽

旁观者设计要点：
- 选择="跟谁走"——每次跟随一个角色经历一段故事
- 不同角色视角揭示不同信息碎片
- 最终需要综合多个视角才能推理出真相（垂直钻探式体验）

**输出**：`.interactive-state.json` + 项目概要 + 项目目录结构初始化

**大规模项目初始化**（15+ 决策点 / 50+ 节点）：
自动生成工程化目录结构（参照 `references/scale-engineering.md`）：
- `story-bible.md` — 全局设定（始终加载的固定上下文）
- `manifest.yaml` — 节点索引 + 依赖图
- `variables.yaml` — 状态变量注册表（含 written_by / read_by 追踪）
- `codex/` — 知识条目目录（角色/地点/线索，按需加载）
- `nodes/` — 节点文件目录（一节点一文件，YAML frontmatter）
- `summaries/` — 滚动摘要目录
- `validation/` — 验证报告

### `/branches` — 叙事拓扑与选择架构

> 原 `/branches` 和 `/choices` 合并为一个命令。拓扑和选择是同一个设计问题的两个维度——
> 决策点的类型决定拓扑形状，拓扑形状约束决策点的设计空间。分开做会导致回溯返工。

**加载参考**：`references/branch-architecture.md`, `references/choice-design.md`, `references/advanced-if.md`, `references/scale-engineering.md`

**分两阶段执行，输出合并到一份 `branch-map.md`**：

#### 阶段一：拓扑设计（宏观）

```
一、分支拓扑选择

   可选拓扑模型：

   A. 瓶颈型（Bottleneck）
      分支 → 汇聚 → 分支 → 汇聚 → 结局分化
      适合：中等复杂度，创作成本可控

   B. 平行调查型（Parallel Investigation）
      共同起点 → 多条调查线并行 → 信息汇总节点 → 结局
      适合：侦探类，不同线索链通向不同真相片段

   C. 时间循环型（Time Loop）
      同一事件反复经历 → 每次携带前次信息 → 最终完美通关
      适合：密室、"土拨鼠之日"式悬疑

   D. 信任网络型（Trust Network）
      选择信任/怀疑不同角色 → 获取不同信息 → 影响同盟和结局
      适合：社会派推理，角色关系驱动

   E. 自由探索型（Open Investigation）
      场景/线索节点自由访问 → 收集足够信息后触发结局 → 结局品质取决于信息量
      适合：经典解谜，高自由度

二、关键节点图

   [Mermaid 流程图：展示所有分支节点、汇聚点、结局点]

三、分支统计
   - 总节点数：N
   - 决策点数：N
   - 唯一路径数：N
   - 最短/最长路径：N / N 个节点
   - 总文本量估算：N 字

四、决策点类型分配
   | 决策点 | 拓扑位置 | 选择类型 | 选项数 | 后果模式 |
   |--------|---------|---------|--------|---------|
   | D01    | 第一分支 | 信息选择 | 3      | 路线分化 |
   | D02    | 中段    | 信任选择 | 2      | 状态变化 |
   | D03    | 汇聚后  | 道德选择 | 2      | 结局影响 |
   | ...    |         |         |        |         |

   选择类型分布检查：
   □ 不超过 50% 的决策点是同一类型
   □ 高风险选择不集中在前半段（前期应让读者积累信息和信心）
   □ 道德/信任选择在角色关系建立之后出现
```

#### 阶段二：逐点选择设计（微观）

在同一份 `branch-map.md` 中，紧接拓扑设计之后，逐个细化每个决策点：

```
五、决策点详细设计

---
### 决策点 D01：{名称}

上下文：
  - 叙事状态：{描述}
  - 读者已知信息：{列表}
  - 情感温度：{低/中/高}
  - 选择类型：{策略/道德/性格/信息/风险/时间压力/叙事}

选项设计：

  选项 A：{文案}
    - 读者选择动机：{为什么有人会选这个}
    - 隐含信息：{这个选择暗示读者的推理倾向}
    - 导向：{下一个节点 ID}
    - 状态变更：{变量变化}
    - 后果预览等级：{明确 / 模糊 / 隐藏}

  选项 B：{文案}
    - （同上结构）

  选项 C（可选）：{文案}
    - （同上结构）

质量检查：
  □ 没有"正确答案"——每个选项都有合理理由
  □ 选项之间有真实的取舍（trade-off）
  □ 选项反映不同的价值观或推理策略
  □ 选项文案不会泄露后续剧情
  □ 不存在"陷阱选项"（看起来合理实则必死）
  □ 每个选项预期选择率 ≥ 20%
  □ 选项后的内容量差异不超过 2 倍

---
### 决策点 D02：{名称}
（同上结构，逐点展开）
```

**选择设计原则**（全局适用）：
- **两难优于多选**：2-3 个选项最佳，超过 4 个造成决策瘫痪
- **后果有延迟**：最好的选择在做出时看不出好坏，数个节点后才显现影响
- **情感与逻辑并重**：理性最优解 ≠ 情感最优解时，产生最好的选择体验
- **"可辩护"原则**：每个选项都应该能被合理辩护，没有"蠢选项"
- **类型驱动拓扑**：选择类型（信息/信任/风险）决定了该节点的分支形状和后果模式

### `/node {编号}` — 逐节点创作

**加载参考**：`references/choice-design.md`, `references/state-management.md`, `references/scale-engineering.md`

**上下文注入协议**（大规模项目必须遵守）：
```
写作节点 N 时，加载以下上下文（按优先级）：

固定层（始终加载）：
  1. story-bible.md（全局设定精简版）
  2. variables.yaml（状态变量定义）
  3. manifest.yaml 中本节点的元数据

动态层（按需加载）：
  4. codex/ 中本节点 codex_links 引用的条目
  5. summaries/running-summary.md（全局滚动摘要）
  6. from[] 节点中最近的一个（完整文本）
  7. 同章节其他节点的 1 句话摘要

绝不加载：
  ✗ 所有节点的完整文本
  ✗ 与本节点无关的 codex 条目
  ✗ 已消解的红鲱鱼详情
```

**输出格式**（生成 `nodes/{chapter}/n{NNN}-{slug}.md`）：

```yaml
---
id: n{NNN}
title: {标题}
chapter: {章节号}
type: narrative | decision | convergence | ending
from: [{来源节点ID列表}]
to: [{后续节点ID列表}]
reads: [{读取的状态变量}]
writes: [{修改的状态变量}]
conditions: "{进入条件表达式}"
codex_links: [{关联的知识条目ID}]
word_count: {实际字数}
status: draft | review | final
---

# {标题}

{正文：800-2000字}

## 选择（如果是决策节点）

> **{选项 A 文案}**
> → n{目标} ({状态变更})

> **{选项 B 文案}**
> → n{目标} ({状态变更})
```

**写完后必须执行**：
1. 更新 `summaries/` 中本章摘要
2. 更新 `codex/` 中发生状态变化的角色/线索条目
3. 更新 `manifest.yaml` 中本节点的 status 和 word_count

### `/consistency` — 全局审查

**加载参考**：`references/narrative-convergence.md`, `references/scale-engineering.md`

**执行逻辑**（两阶段）：

**阶段 1：结构验证（只读 frontmatter，不读正文——省上下文）**

解析所有节点的 YAML frontmatter + manifest.yaml + variables.yaml：

```
一、路径完整性
   □ 所有节点从起始节点可达（图遍历）
   □ 所有非结局节点有至少一个后续（无死胡同）
   □ 不存在孤立节点子集
   □ 每个结局至少有一条路径可达

二、状态一致性
   □ 所有 reads 中引用的变量在 variables.yaml 中存在
   □ 所有 reads 之前有对应的 writes（沿路径追溯）
   □ 每个 conditions 至少在一条路径上可满足
   □ 数值变量不会超出定义的 range
   □ 不存在矛盾的状态组合

三、涟漪分析
   输出每个变量的 writers→readers 依赖链
   标记"修改此节点会影响哪些下游节点"
```

**阶段 2：叙事验证（按路径加载正文——逐路径检查）**

```
四、关键路径通读（最少 5 条路径）
   - 最短路径 / 最长路径 / 最佳结局 / 最差结局 / 随机路径
   - 每条路径检查：叙事连贯、角色一致、信息无矛盾

五、汇聚节点兼容性
   - 对每个 convergence 类型节点，检查所有 from[] 路径到达时的体验

六、线索公平性（悬疑特有）
   - 每个结局所需的关键线索，是否在该路径上可获取？
   - 是否存在"读者做了正确选择却无法获得真相"的情况？

五、平衡性
   - 各路线的内容量是否大致均衡？
   - 是否存在"最优路线"使其他路线失去吸引力？
   - 不同结局的品质差异是否会让部分读者感到被惩罚？
```

## 从线性小说到互动小说的转化工作流

当输入源是已完成的线性悬疑小说时：

```
步骤 1：分析原作结构
  - 识别原作中的关键决策时刻（角色做出选择的地方）
  - 标记"如果ta做了不同选择会怎样"的分歧点
  - 识别读者会想"我会怎么做"的时刻

步骤 2：选择互动化策略
  - 策略 A：决策点互动化（在角色决策处给读者选择权）
  - 策略 B：视角互动化（读者选择跟随哪个角色）
  - 策略 C：信息互动化（读者选择调查哪条线索）
  - 策略 D：混合策略

步骤 3：设计分支差异
  - 最小差异：同一结局不同路径
  - 中等差异：次要情节不同，主线不变
  - 最大差异：不同结局，完全分化的第三幕

步骤 4：处理原作内容
  - 保留：可直接复用的叙事段落（通常 30-50%）
  - 改写：需要适配分支上下文的段落（通常 20-30%）
  - 新增：分支独有的新内容（通常 20-30%）
  - 删除：线性叙事中的过渡，互动版中由选择替代（通常 10-20%）

步骤 5：处理揭示场景的多路径适配
  - 原作的单一揭示 → 根据读者信息状态分化为多个揭示变体
  - 完全信息路径：最接近原作的揭示（完美结局）
  - 部分信息路径：简化版揭示 + 遗留谜团（好结局）
  - 缺失信息路径：错误推理 / 无法揭示（差结局）
```

| `structure.md` → 悬念管理表 | 各路线的信息释放必须覆盖这些悬念 |
| `chapters/ch{N}.md` → 正文 | 保留/改写/拆分为节点 |
| `chapters/ch{N}.md` → 创作备注 | 线索追踪 → 状态变量设置 |

## 工作流程序

```
/init → /branches → /character-paths → /state
                                            ↓
        /export ← /consistency ← /ending N..1 ← /node 1..N
                                  ↑              |
                                  └── 迭代修改 ──┘

注：/branches 包含拓扑设计（阶段一）和逐点选择设计（阶段二），
    原 /choices 已合并进 /branches。
```
