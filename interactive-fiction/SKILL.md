# 互动悬疑小说创作系统 — Interactive Thriller Fiction

> 将悬疑小说转化为带分支选择的互动叙事体验
> 融合 Choice Architecture（选择架构）、Narrative Design（叙事设计）、Game Writing（游戏写作）方法论

---

## 系统定位

本技能将线性悬疑小说转化为**多分支互动小说**，读者在关键时刻做出选择，影响调查方向、角色命运和最终结局。适用于：

- 互动小说平台（Twine / ink / ChoiceScript / 橙光 / 快点阅读）
- 互动影视脚本（Netflix 交互式 / 短剧互动版）
- 文字冒险游戏
- EPUB 互动增强版

---

## 核心理念

### 三大设计支柱

1. **有意义的选择**：每个分支选择必须让读者感受到"我的决定改变了什么"
2. **叙事完整性**：无论走哪条路线，故事都必须在逻辑和情感上完整
3. **可控复杂度**：分支数量在创作可行性范围内，避免指数爆炸

### 悬疑类型的特殊优势

悬疑是最适合互动化的文学类型：
- **调查天然是选择**：先查哪条线索？先审谁？
- **信息即权力**：不同路线获取不同信息，影响推理
- **多结局有天然锚点**：真凶已定，但读者能否找到 ta？
- **失败有叙事价值**：推理错误不是"game over"而是"另一个故事"

**输入要求**：

| 参数 | 说明 | 选项 |
|------|------|------|

### `/分支架构` — 叙事拓扑设计

**加载参考**：`references/branch-architecture.md`

**输出结构**（生成 `branch-map.md`）：

```
一、分支拓扑选择

   可选拓扑模型：

   A. 瓶颈型（Bottleneck）
      分支 → 汇聚 → 分支 → 汇聚 → 结局分化
      适合：中等复杂度，创作成本可控

   B. 平行调查型（Parallel Investigation）
      共同起点 → 多条调查线并行 → 信息汇总节点 → 结局
      适合：侦探类，不同线索链通向不同真相片段

   C. 时间循环型（Time Loop）
      同一事件反复经历 → 每次携带前次信息 → 最终完美通关
      适合：密室、"土拨鼠之日"式悬疑

   D. 信任网络型（Trust Network）
      选择信任/怀疑不同角色 → 获取不同信息 → 影响同盟和结局
      适合：社会派推理，角色关系驱动

   E. 自由探索型（Open Investigation）
      场景/线索节点自由访问 → 收集足够信息后触发结局 → 结局品质取决于信息量
      适合：经典解谜，高自由度

二、关键节点图

   [Mermaid 流程图：展示所有分支节点、汇聚点、结局点]

三、分支统计
   - 总节点数：N
   - 决策点数：N
   - 唯一路径数：N
   - 最短路径：N 个节点
   - 最长路径：N 个节点
   - 总文本量估算：N 字
```

### `/选择设计` — 选择点设计

**加载参考**：`references/choice-design.md`

**为每个决策点输出**（生成 `choices/decision-{NNN}.md`）：

```
决策点 #{N}：{名称}

一、上下文
   - 此刻的叙事状态
   - 读者已知信息
   - 情感温度

二、选项设计
   选项 A：{文案}
   - 读者选择动机：{为什么有人会选这个}
   - 隐含信息：{这个选择暗示读者的推理倾向}
   - 导向：{下一个节点}
   - 后果预览等级：{明确 / 模糊 / 隐藏}

   选项 B：{文案}
   - （同上结构）

   选项 C（可选）：{文案}
   - （同上结构）

三、选择质量检查
   □ 没有"正确答案"——每个选项都有合理理由
   □ 选项之间有真实的取舍（trade-off）
   □ 选项反映不同的价值观或推理策略
   □ 选项文案不会泄露后续剧情
   □ 不存在"陷阱选项"（看起来合理实则必死）
```

**选择设计原则**：
- **两难优于多选**：2-3 个选项最佳，超过 4 个造成决策瘫痪
- **后果有延迟**：最好的选择在做出时看不出好坏，数个节点后才显现影响
- **情感与逻辑并重**：理性最优解 ≠ 情感最优解时，产生最好的选择体验
- **"可辩护"原则**：每个选项都应该能被合理辩护，没有"蠢选项"

**输出**（生成 `character-branches.md`）：

```

|----------|-------------|-------------|-------------|
| 决策 #3  | 信任增加    | 信任降低    | 关系中立    |
| 决策 #7  | 存活       | 死亡        | 受伤        |
| ...      | ...        | ...         | ...         |

角色弧线变体：
- 路线 α（信任路线）：{角色弧线描述}
- 路线 β（怀疑路线）：{角色弧线描述}
- 路线 γ（中立路线）：{角色弧线描述}

跨路线一致性要求：
- 核心性格特质在所有路线中保持不变
- 动机在所有路线中自洽
- 差异来自"不同压力下的不同反应"，而非"不同人格"
```

### `/状态系统` — 游戏状态设计

**加载参考**：`references/state-management.md`

**输出**（生成 `state-system.md`）：

```
一、状态变量清单

   数值型变量：
   | 变量名 | 说明 | 初始值 | 范围 | 影响 |
   |--------|------|--------|------|------|
   | trust_{角色} | 对角色的信任度 | 50 | 0-100 | 信息获取、结局 |
   | evidence_count | 已收集证据数 | 0 | 0-N | 结局品质 |
   | suspicion_level | 犯人对侦探的警觉 | 0 | 0-100 | 难度、危险事件 |

   布尔型变量：
   | 变量名 | 说明 | 触发条件 | 影响 |
   |--------|------|----------|------|
   | found_{线索ID} | 是否发现特定线索 | 特定节点选择 | 推理选项解锁 |
   | alive_{角色} | 角色是否存活 | 关键决策 | 后续剧情 |

二、结局触发条件
   | 结局 | 名称 | 触发条件 | 品质评级 |
   |------|------|----------|----------|
   | E1 | 完美破案 | evidence ≥ 8 AND alive_all AND trust_key ≥ 70 | ★★★★★ |
   | E2 | 真相大白但有牺牲 | evidence ≥ 6 AND !alive_{某角色} | ★★★★ |
   | E3 | 冤案 | evidence < 4 OR wrong_accusation | ★★ |
   | E4 | 侦探陨落 | suspicion ≥ 90 AND !escape_route | ★ |

三、状态可视化建议
   - 哪些状态对读者可见（增强策略感）
   - 哪些状态隐藏（增强悬疑感）
```

### `/一致性检查` — 全局审查

**加载参考**：`references/narrative-convergence.md`

**执行逻辑**：

```
一、路径完整性
   - 所有节点是否可达？
   - 是否存在死胡同（无后续节点的非结局节点）？
   - 是否存在孤立节点？

二、状态一致性
   - 所有使用的状态变量是否已定义？
   - 状态检查条件是否可能为真？
   - 是否存在不可能到达的结局？

三、叙事连贯性
   - 从每条可能路径阅读，故事是否通顺？
   - 汇聚节点是否能兼容所有来源路径？
   - 角色是否在不同路径中行为自洽？

四、线索公平性（悬疑特有）
   - 每个结局所需的关键线索，是否在该路径上可获取？
   - 是否存在"读者做了正确选择却无法获得真相"的情况？

五、平衡性
   - 各路线的内容量是否大致均衡？
   - 是否存在"最优路线"使其他路线失去吸引力？
   - 不同结局的品质差异是否会让部分读者感到被惩罚？
```

## 从线性小说到互动小说的转化工作流

当输入源是已完成的线性悬疑小说时：

```
步骤 1：分析原作结构
  - 识别原作中的关键决策时刻（角色做出选择的地方）
  - 标记"如果ta做了不同选择会怎样"的分歧点
  - 识别读者会想"我会怎么做"的时刻

步骤 2：选择互动化策略
  - 策略 A：决策点互动化（在角色决策处给读者选择权）
  - 策略 B：视角互动化（读者选择跟随哪个角色）
  - 策略 C：信息互动化（读者选择调查哪条线索）
  - 策略 D：混合策略

步骤 3：设计分支差异
  - 最小差异：同一结局不同路径
  - 中等差异：次要情节不同，主线不变
  - 最大差异：不同结局，完全分化的第三幕

步骤 4：处理原作内容
  - 保留：可直接复用的叙事段落
  - 改写：需要适配分支上下文的段落
  - 新增：分支独有的新内容
  - 删除：线性叙事中的过渡，互动版中由选择替代
```
