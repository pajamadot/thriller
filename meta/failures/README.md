# 故事失败档案 — Story Failure Archive

> 方法论的 graveyard.md 记录被移除的**规则**；这里记录被拒绝的**生成结果**。
> 格式参考 RLHF（Reinforcement Learning from Human Feedback）的偏好数据集：
> 每条记录包含 reject（失败样本）+ 原因分析 + accept（修正方向）+ 约束溯源。

---

## 设计理念

```
传统方法论进化：规则 → 实践 → 失败 → 修规则
                                       ↑
                               但失败的具体样本被丢弃了

RLHF 式进化：  规则 → 实践 → 失败 → 保留失败样本 + 标注原因 → 修规则
                                                                    ↑
                                         AI 可以从具体的失败中学习模式
```

**为什么保留失败样本比只保留规则更有价值**：
- 规则告诉你"不要做X"，但不告诉你"X 长什么样"
- 失败样本让 AI 看到具体的错误模式，建立直觉
- 对比 reject/accept 对，AI 学到的不是抽象原则而是具体判断
- 多个失败样本可以暴露规则未覆盖的边缘情况

## 记录格式

每个失败案例一个文件，命名：`{序号}-{项目名}-{失败类型简述}.md`

例：`001-locked-room-gathering-motivation.md`

### 文件模板

```markdown

- **失败类型**: {约束真空 / 逻辑矛盾 / 动机不足 / 时间线冲突 / 人物工具化 / ...}
- **严重性**: {致命(需要重写) / 严重(需要大幅修改) / 中等(可补丁但有代价)}
```

## 失败类型分类法

### 约束层失败

| 类型 | 定义 | 典型信号 |
|------|------|---------|
| 约束真空 | 某个必要约束没有被任何步骤建立 | 生成时"临时发明"关键信息 |
| 约束冲突 | 两个步骤建立了互相矛盾的约束 | 补丁A修好了X但破坏了Y |
| 约束传递失败 | 约束在上游建立了但下游未使用 | 设计文档中有但生成文本中丢失 |
| 约束过度 | 约束太多导致无法同时满足 | 生成陷入死锁，每个方向都违反某条约束 |

### 内容层失败

| 类型 | 定义 | 典型信号 |
|------|------|---------|
| 动机不足 | 角色的行为缺乏可信的内在驱动力 | "为什么ta要这么做？"无法自圆其说 |
| 逻辑矛盾 | 故事内部的事实互相冲突 | 时间线、空间、物理可行性漏洞 |
| 人物工具化 | 角色只为情节服务，没有独立的存在价值 | 移除角色后故事不受影响 |
| 信息泄露 | 叙事控制失败，过早暴露了不该暴露的信息 | 读者在中点前已能确定真凶 |
| 情感断裂 | 角色的情感反应与情境不匹配 | "这个人不应该这么平静/激动" |
| 节奏失控 | 信息密度或事件密度不合理 | 单章塞了太多线索，或连续3章无进展 |

### 结构层失败

| 类型 | 定义 | 典型信号 |
|------|------|---------|
| 承重缺失 | 关键的叙事承重点没有被建立 | 揭示缺乏铺垫，反转没有伏笔 |
| 死端路径 | 分支叙事走入死胡同 | 互动叙事中某条路径无法收束 |
| 比例失衡 | 三幕/章节比例严重偏离目标 | 第一幕占40%，第三幕只有10% |

## 使用指南

### 何时记录

```
必须记录：
  ✓ 生成后发现需要重写的内容（严重性：致命/严重）
  ✓ 通过对话补丁修复但产生连锁问题的情况
  ✓ 约束饱和检查失败的案例
  ✓ /check 评分低于 30 分的章节

建议记录：
  ~ /check 评分 30-40 的章节（如果能识别出具体的方法论原因）
  ~ 虽然最终修好了但暴露了方法论盲区的情况

不需要记录：
  ✗ 纯文风/措辞层面的问题（属于执行层面，不反映方法论缺陷）
  ✗ 篇幅控制问题（字数多了/少了）
  ✗ 一次性的偶发问题，无法泛化
```

### 如何使用

```
1. 新项目启动时
   浏览失败档案，检查：
   - 同子类型的历史失败 → 本项目是否有相同风险？
   - 同失败类型的模式 → 约束饱和检查是否已覆盖？

2. /retro 阶段
   将本项目的失败案例记录到档案
   与历史失败交叉对比，识别反复出现的模式

3. /evolve 阶段
   统计失败类型分布 → 最高频的类型 = 方法论最需要加强的方向
   统计失败的约束溯源 → 最常被溯源到的步骤 = 该步骤的契约需要更新

4. /blindspot 阶段
   检查：有没有失败类型在分类法中没有对应的类别？
   → 如果有 → 扩展分类法，可能暴露了新的盲区
```

### 与其他系统的关系

```
失败档案 → 约束契约：失败暴露的约束真空 → 更新契约的 ⚠/✓
失败档案 → graveyard.md：失败导致移除的规则 → 记录在墓地
失败档案 → /evolve：反复出现的失败模式 → 触发方法论进化
失败档案 → /blindspot：未分类的失败 → 可能是新的盲区类型
失败档案 → 步骤执行日志：失败的根因 → 追溯到哪个步骤的哪个决策
```

## 统计仪表盘

定期（每 3-5 个项目）统计：

```
失败频率分布：
  | 失败类型 | 次数 | 趋势(↑/↓/→) | 是否已有防护 |

约束溯源热力图：
  | 步骤 | 被溯源次数 | 最常见的约束真空 |

发现方式分布：
  | 发现方式 | 次数 | 占比 |
  → 目标：约束饱和检查发现的比例应持续上升（前置拦截率提高）

修复效果追踪：
  | 防护措施 | 加入日期 | 之后的项目中是否再出现同类失败 |
  → 如果防护措施加入后仍出现 → 防护不充分 → 需要加强
```
