# 认知盲区检测

> 方法论最危险的问题不是"错误的规则"，而是"从未被提出的问题"
> 参考 Charlie Munger 的"反向思维"、Nassim Taleb 的"反脆弱"思想

---

## 一、五种盲区类型

### 1. 继承盲区（Inherited Blind Spots）

```
来源：从权威（McKee、Hitchcock、Knox...）继承的观点，从未被质疑

检测方法：
  对方法论中每一条标注了"引自 XXX"的规则，问：
  - 这条规则是在什么语境下提出的？
  - 那个语境与我们的语境（AI辅助创作、中文悬疑、互动叙事）相同吗？
  - 有没有成功违反这条规则的作品？
  - 如果这条规则不存在，我们会怎么做？

高危继承盲区：
  - Knox 十诫写于 1929 年，近百年前的文化语境
  - Syd Field 的三幕结构理论主要基于好莱坞电影
  - McKee 的场景理论基于西方戏剧传统
  → 这些理论在中文网文/短剧/互动叙事中是否仍然完全适用？
```

### 2. 幸存者偏差盲区（Survivorship Bias）

```
来源：我们只研究了成功的作品，忽略了使用相同方法但失败的作品

检测方法：
  - 是否存在严格遵循我们方法论但仍然失败的案例？
  - 方法论中"必须"的规则，有多少是从成功案例中归纳的？
  - 那些成功案例是因为遵循了这条规则而成功的吗？还是因为其他原因？

例：
  "Christie 的嫌疑人矩阵通常有 5-8 人 → 方法论建议 5-8 人"
  但 Christie 也有失败/平庸的作品使用了同样的嫌疑人数量。
  嫌疑人数量可能不是成功的因果因素。
```

### 3. 锚定盲区（Anchoring Bias）

```
来源：方法论的第一个版本设定了"锚点"，后续迭代围绕锚点微调而非跳出

检测方法：
  - 如果今天从零开始设计这个方法论，最终结果会和现在一样吗？
  - 哪些结构是"因为一开始就这样设计"而保留的？
  - 有没有被锁定在次优状态的模块？

练习：
  随机选择方法论的一个模块（如"角色原型"），
  假设它完全不存在，问"解决同一问题我会怎么设计？"
  如果新设计明显不同 → 可能存在锚定盲区
```

### 4. 可用性盲区（Availability Bias）

```
来源：最近的项目经验权重过高，遥远的/未经历的情况被忽视

检测方法：
  - 最近的修改是否过度特化于最近的项目类型？
  - 方法论是否在某个子类型上过度详细，在另一个上过度粗糙？
  - 有没有子类型从未被用来创作？那个子类型的指导够用吗？

例：
  如果连续做了 3 个心理悬疑项目后进化方法论，
  可能过度优化了心理悬疑的指导，
  而本格推理的指导可能已经过时或不足了。
```

### 5. 专业化盲区（Expert Blind Spot）

```
来源：随着使用方法论的经验增加，不再能看到新手的困难

检测方法：
  - 方法论中的术语对新用户是否友好？
  - 参考资料的信息密度是否过高？
  - 有没有"只有做过 5+ 个项目的人才知道怎么用"的模块？
  - 快速入门路径是否存在？

修复：
  定期让新用户尝试使用方法论，观察ta在哪里卡住
```

## 二、反向思维练习

### "Pre-mortem"法

```
在开始下一个项目之前，假设项目已经失败了。

问：
  "假设这个项目最终的作品质量很差，
   最可能的原因是什么？"

然后检查方法论是否有针对这个原因的防护。

例：
  假设失败原因是"线索设计过于复杂，读者无法追踪"
  → 检查 clue-design.md 是否有线索过载的早期预警机制
  → 如果没有 → 这是一个盲区
```

### "邪恶对手"法

```
假设有一个恶意的创作者，想要用我们的方法论写出最糟糕的作品。
ta 会怎么利用方法论的漏洞？

例：
  - 严格按照线索比例，但线索都没有叙事意义 → 方法论是否只管数量不管质量？
  - 严格按照三幕结构，但每幕内部没有变化 → 方法论是否过于关注宏观忽略微观？
  - 完美的嫌疑人矩阵，但每个角色都像纸片人 → 方法论是否分离了技术和人文？

这些漏洞就是盲区。
```

### "时间旅行"法

```
假设 5 年后回看现在的方法论，哪些部分会显得过时？

考虑：
  - AI 写作工具的发展
  - 读者注意力持续缩短
  - 新的叙事媒介（VR/AR/脑机接口）
  - 文化全球化 vs 文化在地化
  - 悬疑类型本身的变异

标记那些最可能过时的部分 → 增加灵活性或提前准备替代方案
```

## 三、盲区检测检查清单

### 每次执行 `/blindspot` 时遍历

```
□ 是否存在从未被用过的参考资料？
  → 如果有：是缺少使用场景，还是内容本身没有价值？

□ 是否存在从未被质疑过的"铁律"？
  → 列出所有"必须/禁止"的规则，逐条问"如果违反会怎样？"

□ 是否存在过于复杂但很少使用的模块？
  → 复杂度与使用频率不匹配可能意味着设计过度

□ 方法论的目标用户画像是否更新？
  → 用户可能在成长——新手变老手，方法论是否跟上？

□ 是否存在未定义的术语或概念？
  → 术语在多次迭代中可能漂移含义

□ 方法论中的数字（比例、数量建议）是否有依据？
  → "5-8个嫌疑人" — 这个数字来自哪里？还适用吗？

□ 是否存在互相矛盾的建议？
  → 检查不同参考资料中对同一问题的建议是否一致

□ 最近的进化是否让方法论变得更复杂了？
  → 如果是：复杂度是否带来了等比例的价值？
```

## 四、盲区检测的输出

```
每次 /blindspot 扫描输出：

一、发现的盲区
  | 盲区 | 类型 | 严重性 | 可能的影响 |

二、打破的假设
  | 假设 | 原始来源 | 质疑理由 | 建议行动 |

三、进化方向建议
  基于盲区分析，推荐的 1-3 个进化方向

四、保持的信念
  经过质疑后仍然成立的核心原理（增强置信度）
```
