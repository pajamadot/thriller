# 方法论进化模式

> 方法论如何从"有用"变得"更有用"的系统化路径
> 参考进化论思维、科学方法、精益创业的 Build-Measure-Learn 循环

---

## 一、进化的四种模式

### 模式 1：填充（Gap Filling）

```
信号：创作过程中遇到方法论没有覆盖的情况
性质：方法论的知识面不够广

例：
  "写密室推理时发现参考资料中没有不可能犯罪的构造方法"
  → 需要新增 references/impossible-crime.md

行动：
  1. 记录缺失领域
  2. 研究该领域的已有知识
  3. 创建新的参考资料
  4. 在 SKILL.md 中增加对应的命令/流程引用

风险：低（纯增量，不改变已有内容）
频率：最常见，几乎每个项目都会触发
```

### 模式 2：修正（Correction）

```
信号：方法论的某条规则/建议在实践中被证明不准确
性质：方法论的观点有误

例：
  "方法论建议'红鲱鱼数量 ≤ 关键线索数量'，
   但在复杂长篇中，2倍的红鲱鱼反而效果更好"
  → 需要修正 red-herring.md 中的比例建议

行动：
  1. 记录反例及其语境
  2. 分析原始规则的适用范围
  3. 修正为更精确的条件性建议
  4. 添加反例作为注释

风险：中等（修改已有内容，可能影响引用该规则的其他部分）
频率：中等，每 2-3 个项目可能触发一次
```

### 模式 3：重构（Restructuring）

```
信号：方法论的某个模块在整体架构中的位置不对
性质：方法论的结构有问题

例：
  "pacing-tension.md 和 suspense-technique.md 有大量重叠，
   每次使用都要同时参考两个文件且找不到需要的内容"
  → 需要重新划分两个文件的职责边界

行动：
  1. 明确问题：哪些用户场景受影响
  2. 设计新结构：重新划分职责
  3. 迁移内容：按新结构重组
  4. 验证：确认常见用户场景在新结构下更流畅

风险：高（影响面广，可能引入新问题）
频率：低，每 5+ 个项目可能触发一次
```

### 模式 4：蜕变（Metamorphosis）

```
信号：方法论的基本假设被颠覆
性质：方法论的范式需要更新

例：
  "随着AI辅助创作的普及，'侦探角色'可能不再是必需的——
   读者自己就是侦探。方法论的整个角色体系需要重新思考"
  → 需要在角色原型系统中引入'无侦探叙事'的可能性

行动：
  1. 承认范式变化
  2. 分析新范式下方法论哪些仍然有效
  3. 识别需要重新设计的模块
  4. 分阶段替换，保持系统可用性

风险：极高（可能动摇方法论基础）
频率：极低，可能数年一遇
频率信号：当 /盲区 扫描连续 2 次指向同一根本性问题时
```

### 模式 5：接口修复（Interface Repair）

```
信号：反复出现"临时发明"或"事后补丁"，但每个步骤独立看没有问题
性质：步骤之间的约束传递有缺口——问题在接缝处，不在组件内

区分于其他模式：
  - 填充 = 缺少知识 → 加知识
  - 修正 = 知识有误 → 改知识
  - 重构 = 知识位置不对 → 移知识
  - 蜕变 = 知识范式过时 → 换框架
  - 接口修复 = 知识都在，但传递失败 → 补接缝

例：
  "密室故事中人物到场动机不足。/trick 正确，/characters 正确，
   但两者之间没有步骤负责建立'为什么大家在一起'的约束。"
  → 需要新增 /premise 步骤或将聚合前提归入现有步骤

识别方法：
  1. 检查 /retro 接口审计中的"临时发明"记录
  2. 如果同类临时发明在 2+ 个项目中出现 → 接口问题
  3. 对照 constraint-contracts.md 的约束契约
  4. 定位哪个 ⚠ 项没有对应的 ✓ 覆盖

行动：
  1. 在约束契约中将真空约束显式化
  2. 决定归属方式：
     A. 新增步骤（约束复杂度足够、跨多个现有步骤）
     B. 扩展现有步骤的输出（约束自然属于某个步骤但未被包含）
     C. 新增跨步骤检查点（约束简单但需要多步骤协同验证）
  3. 更新约束契约的 ✓/←/⚠ 清单
  4. 在下一个项目中执行约束饱和检查验证

风险：中等（不改变步骤内容，只改变步骤边界和传递关系）
频率：中等，每 2-4 个项目可能触发一次
频率信号：/retro 接口审计中"临时发明"项 ≥ 2，或补丁率 > 10%
```

## 二、进化决策框架

### 什么时候改 vs 什么时候不改

```
应该改的信号：
  ✓ 连续 2+ 个项目在同一个点遇到问题
  ✓ 问题被归因到方法论层面（不是执行或能力）
  ✓ 有明确的修改方案（不是"感觉不好"）
  ✓ 修改后有可验证的改善预期

不应该改的信号：
  ✗ 只有 1 个项目遇到了这个问题
  ✗ 问题可以通过更好的执行来解决
  ✗ "这个规则太严了"（严格的规则存在是有原因的）
  ✗ 受到了某本新书/课程的影响但没有实践验证
  ✗ 改了之后不知道怎么验证效果

灰区处理：
  标记为"观察中"，在下 2 个项目中刻意关注
  如果问题再次出现 → 改
  如果问题不再出现 → 保持
```

### 改多少的原则

```
最小可行修改（Minimum Viable Change）：

  不要一次改太多。
  每次进化迭代最多修改 3 个具体点。
  每个修改必须独立（A 的效果不依赖于 B 是否也改了）。

  原因：
  - 如果同时改了 5 个点，效果变好了，不知道是哪个改动的功劳
  - 如果效果变差了，不知道是哪个改动出了问题
  - 小步迭代可以更精确地归因
```

## 三、进化日志格式

### 每次进化的标准记录

```markdown
# Evolution v{N}

## 触发项目
{项目名列表}

## 模式
{填充 / 修正 / 重构 / 蜕变}

## 变更

### 变更 1: {标题}
- **假设**: {如果改X为Y，则Z会改善}
- **变更前**: {原文/原结构}
- **变更后**: {新文/新结构}
- **影响文件**: {文件列表}
- **验证标准**: {怎么知道这个改动有效}
- **状态**: 待验证 / 已验证有效 / 已验证无效 / 已回滚

## 待验证清单
在下一个项目中关注：
- [ ] 假设 1 是否生效？
- [ ] 是否引入了新问题？
```

## 四、知识的三种状态

### 方法论中的每条知识都处于三种状态之一

```
1. 原理（Principle）
   经过多次验证的稳定知识
   例："每个场景必须有信息状态变化"
   标记：无特殊标记（默认所有内容都应该是原理级别）
   改动条件：极高证据标准

2. 实践（Practice）
   基于经验但可能有适用范围的建议
   例："红鲱鱼数量不超过关键线索数量"
   标记：可在末尾标注 [经验性建议]
   改动条件：中等证据标准

3. 实验（Experiment）
   刚加入的、尚未经过充分验证的新知识
   例：新增的某个技术或某条规则
   标记：[进化 v{N} | 待验证]
   改动条件：低——随时可调整

知识的生命周期：
  新增时 → 实验
  1-2 个项目验证后 → 实践
  5+ 个项目验证后 → 原理
```

## 五、防退化机制

### 确保进化不会引入新问题

```
1. 回归测试
   每次修改后，用最近 2 个项目的经验"虚拟回测"：
   "如果当时用了新版方法论，结果会更好还是更差？"

2. 回滚协议
   如果新版方法论在下一个项目中导致了
   之前不存在的问题 → 立即回滚该修改

3. Graveyard 审计
   每 5 次进化后，回顾 graveyard.md：
   被删除的规则中有没有需要重新启用的？
   （有时候删除一个规则的原因已经不存在了）

4. 复杂度预算
   方法论的总文件大小有上限。
   每新增一段内容，考虑是否可以删除或合并已有内容。
   方法论应该越来越精炼，而非越来越臃肿。
```

## 六、进化的哲学观

```
好的方法论像好的免疫系统：
  - 遇到新病原体（新的创作挑战）后会记住应对方式
  - 不会过度反应（不因一次失败就推翻整个系统）
  - 会区分自我和非自我（知道什么是核心原理、什么可以变）
  - 有时候需要忍受暂时的不适来获得长期的免疫力

而不是像官僚系统：
  - 每次出问题就加一条新规则
  - 规则越来越多，系统越来越僵化
  - 最终规则之间互相矛盾
  - 没有人记得每条规则的原始理由
```
