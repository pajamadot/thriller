# 系统评价体系 — Evaluation Framework

> 三阶段全覆盖：设计评价 → 生成评价 → 项目评价
> 目标：让逻辑漏洞在设计阶段被拦截，而非在生成后被发现

---

## 一、评价体系总览

```
                    设计阶段                    生成阶段                项目阶段
               ┌─────────────┐          ┌──────────────┐        ┌────────────┐
               │ 文档质量评分  │          │ 章节质量评分   │        │ 项目总评    │
  /theme       │ (5维 × N文档) │  /scene  │ (7维 × N章)   │ /retro │ (5维总评)   │
  /trick  ───► │ 跨文档一致性  │ ───────► │ 约束遵守检测   │ ────► │ 失败模式提取 │
  /premise     │ 约束饱和检查  │          │ 临时发明检测   │       │ 进化信号检测 │
  /characters  │ ═══════════  │          │ 漂移累积追踪   │       │ 方法论评分   │
  /structure   │ 设计就绪度    │          │ ═══════════   │       │ ════════    │
               └──────┬───────┘          └──────┬────────┘       └──────┬─────┘
                      │                         │                       │
                 通过才能开始                 每章执行                  项目结束
                 /scene 生成               累积到项目评价             喂入进化系统
```

## 二、设计阶段评价

### A. 文档质量评分

每份设计文档按 5 个维度评分（1-5 分），总分 25 分。

#### 通用维度

| 维度 | 1分(缺失) | 3分(合格) | 5分(优秀) |
|------|----------|----------|----------|
| **完整性** | 缺少必要输出项 | 所有必要项都有但部分粗糙 | 所有输出项完整且有深度 |
| **内在一致性** | 文档内部有自相矛盾 | 无明显矛盾 | 所有元素互相支撑强化 |
| **可验证性** | 断言无法验证（"角色很有魅力"） | 多数断言可验证 | 每个断言都有具体的验证标准 |
| **约束密度** | 对下游步骤几乎无约束力 | 提供基本约束 | 提供精确约束且不过度约束 |
| **叙事潜力** | 技术合格但缺乏吸引力 | 有趣但不突出 | 读到设计文档就想看故事 |

#### 各文档特化检查

**theme.md 特化检查：**
```
□ 主题前提可辩论（有人会持反对意见）
□ 犯罪是主题的极端表现（而非贴标签式的关联）
□ 反主题有独立说服力（不是稻草人）
□ Egri 三维度各自有至少一个具体的故事化体现
```

**trick-design.md 特化检查：**
```
□ 密室/诡计的物理可行性有量化论证（不是"大概可以"）
□ 线索清单中每条线索有明确的隐蔽手段（不是"自然带出"）
□ 反派引擎 Step 1.5 已完成（目标选择逻辑）
□ 双读者验证有具体的预测（不是泛泛的"惊奇"）
□ 知识状态矩阵至少覆盖前三章
```

**premise.md 特化检查：**
```
□ 每个角色的三层到场动机已填写（独立/关系/代价）
□ 四项反向验证全部执行（减法/替换/时间/拒绝）
□ 聚合事件有独立于犯罪的存在理由
□ 凶手-聚合事件因果链已建立
□ 至少一个"缺席角色"被考虑
```

**characters.md 特化检查：**
```
□ 侦探有超越"解谜"的个人 Stakes
□ 犯人的三层动机完整（表面/真实/根源）
□ 犯人同理性等级 ≥ Level 3
□ 嫌疑人矩阵中每人有独立秘密（与案件无关的个人故事）
□ 角色罗盘（Lack/Yearning/Resistance/Desire）主角和犯人都已完成
□ 关键角色 Agent 卡完整（面具人格 / 真实意图 / 筹码 / 依赖 / 秘密）
□ 关系网络有秘密链（非平面关系）
□ 至少两条关键关系边同时包含资源流和知识不对称
□ 每个角色通过减法测试（移除后故事受影响）
```

**structure.md 特化检查：**
```
□ 三幕比例在合理范围内（第一幕 20-30%，第三幕 20-30%）
□ 中点逆转类型已选择且与子类型适配
□ 章节大纲中每章至少有一次知识状态变化
□ 信息阶梯逐层推进，而非一次性解释
□ 钩子地图已完成：每章回答 1 个问题并打开至少 1 个更贵的问题
□ 悬念管理表无悬念遗忘（每个悬念有揭示章节）
□ MICE 嵌套关闭顺序正确（先开后关）
□ 时间线（故事时间 vs 叙事时间）已对齐
□ 至少一次主要关系翻面由事件触发而非解释触发
```

#### 文档评分汇总

```
| 文档 | 完整性 | 内在一致 | 可验证性 | 约束密度 | 叙事潜力 | 小计 | 特化通过率 |
|------|--------|---------|---------|---------|---------|------|----------|
| theme.md | /5 | /5 | /5 | /5 | /5 | /25 | X/4 |
| trick-design.md | /5 | /5 | /5 | /5 | /5 | /25 | X/5 |
| premise.md | /5 | /5 | /5 | /5 | /5 | /25 | X/5 |
| characters.md | /5 | /5 | /5 | /5 | /5 | /25 | X/9 |
| structure.md | /5 | /5 | /5 | /5 | /5 | /25 | X/8 |

文档质量总分 = 各文档分数之和 / 满分 × 100

评级：
  90-100%：优秀 — 设计扎实，可放心进入生成
  75-89%：良好 — 个别弱项需注意，可进入生成
  60-74%：合格 — 明显弱项需修补后再生成
  <60%：不合格 — 需返工
```

### B. 跨文档一致性验证

文档独立评分可能都合格，但文档之间可能矛盾。这是设计阶段最危险的隐患。

#### 一致性检查矩阵

对每对文档，检查关键交叉约束：

```
theme ↔ trick：
  □ 犯罪机制是否是主题的极端表现？（而非仅仅"与主题有关"）
  □ 揭示时刻是否同时完成诡计揭示和主题论证？
  □ 反主题是否在诡计的"表面叙事"中得到体现？

theme ↔ characters：
  □ 主角弧光是否从反主题到正主题？
  □ 犯人是否体现了主题的极端/扭曲版本？
  □ 每个主要角色是否与主题的某个维度相关？

trick ↔ premise：
  □ 犯罪的空间需求是否被聚合空间满足？
  □ 犯人有犯罪机会吗？（聚合事件是否创造了独处窗口？）
  □ 凶手的目标选择逻辑与聚合人员名单一致吗？

trick ↔ characters：
  □ 犯人的能力是否匹配诡计的技术要求？
  □ 线索清单中的行为线索是否与角色人设一致？
  □ 嫌疑人矩阵中的嫌疑来源是否与线索/红鲱鱼对应？

premise ↔ characters：
  □ 到场动机是否与角色性格一致？（social butterfly 去无信号荒山？）
  □ 关系网络是否支撑"关系理由"层的到场动机？
  □ 角色的秘密是否影响到场意愿？

characters ↔ structure：
  □ 角色弧光是否有足够的章节空间展开？
  □ 嫌疑指数曲线是否与章节大纲的嫌疑转移一致？
  □ 角色退场时机（死亡/消除嫌疑）是否与结构节拍匹配？

premise ↔ structure：
  □ 第一幕的开场逻辑是否自然呈现聚合过程？
  □ 封闭性建立的时间点是否与结构节拍一致？
  □ 封闭后行为逻辑是否在章节大纲中有所体现？

trick ↔ structure：
  □ 线索-章节时间表是否与章节大纲的节拍一致？
  □ 知识状态矩阵的变化节奏是否匹配章节节奏？
  □ 反派引擎的每个 Step 是否有对应的章节承载？
```

#### 一致性评分

```
每对文档检查项：通过=1，不通过=0
一致性分数 = 通过项数 / 总检查项数 × 100%

评级：
  95-100%：无矛盾 — 设计高度统一
  85-94%：基本一致 — 个别不一致需修正
  70-84%：部分矛盾 — 需要协调后再进入生成
  <70%：严重矛盾 — 需要返回设计阶段重做
```

---

### C. 设计就绪度（Design Readiness Score）

在所有设计步骤完成后、第一行正文之前计算：

```
设计就绪度 =
    文档质量总分 × 0.35
  + 跨文档一致性 × 0.35
  + 约束饱和率   × 0.30

评级：
  90-100%：Ready — 直接进入 /scene
  75-89%：Almost Ready — 修补标注的弱项后进入
  60-74%：Needs Work — 返回设计阶段，修复矛盾和真空
  <60%：Not Ready — 重大设计缺陷，需要大幅返工
```

**阻塞规则：**
- 约束饱和率 < 80% → 硬性阻塞，不允许生成
- 任何一份文档质量 < 15/25 → 该文档需返工
- 跨文档一致性 < 70% → 需协调后再生成
- 设计就绪度 < 60% → 阻塞

## 三、生成阶段评价

### A. 增强版章节评分（/check 升级）

在原有 6 维度基础上新增第 7 维度：

| 维度 | 评估要点 | 所有子类型权重 |
|------|---------|-------------|
| 悬念张力 | 信息不对称、阅读欲望 | （按子类型变化） |
| 线索技术 | 线索自然度、误导力 | （按子类型变化） |
| 人物一致性 | 行为/对话辨识度 | （按子类型变化） |
| 节奏控制 | 张弛、场景长度 | （按子类型变化） |
| 逻辑严密性 | 推理链、漏洞 | （按子类型变化） |
| 文学质量 | 风格、意象、金句 | （按子类型变化） |
| **约束遵守** | 设计文档的执行忠实度 | **×1.2（所有子类型）** |

#### 约束遵守维度评分标准

```
1-2分：章节内容与设计文档严重偏离
  - 出现设计文档中未规划的重大情节
  - 角色行为与 characters.md 的人设矛盾
  - 线索操作与 trick-design.md 的计划不符

3-4分：基本遵守，有小幅偏离
  - 多数设计约束被执行
  - 个别地方有"创造性偏离"，偏离合理但未被预规划

5-6分：精确遵守
  - 本章的线索操作与 trick-design 完全一致
  - 嫌疑指向变化与 structure.md 章节大纲一致
  - 角色行为完全在 characters.md 的人设范围内

7-8分：遵守且增强
  - 不仅执行了设计约束，还发现了设计中的优化机会
  - 偏离有明确的叙事收益且不破坏其他约束

9-10分：完美执行 + 涌现
  - 设计约束完美执行
  - 在约束框架内产生了设计文档未预见的优秀叙事
```

### B. 临时发明检测

每章写作完成后，扫描是否出现了**设计文档中没有预规划的信息**：

```
临时发明分类：

  合理发明（Good Improvisation）：
    - 对话中的具体措辞（设计文档不规划具体台词）
    - 环境描写的细节（气味、光线、温度）
    - 角色的微表情和肢体语言
    → 这些是正常的创作空间，不算问题

  可疑发明（Suspicious Improvisation）：
    - 角色透露了设计文档未提及的背景信息
    - 出现了设计文档未规划的物品/地点
    - 角色之间的关系出现了设计文档未定义的维度
    → 需要检查：是否与设计约束矛盾？是否需要回写入设计文档？

  危险发明（Dangerous Improvisation）：
    - 角色的到场动机在本章中被"临时编造"
    - 出现了破坏已建立约束的新信息
    - 时间线或空间逻辑出现了设计文档未覆盖的矛盾
    → 必须处理：修改本章，或修改设计文档（但需检查连锁影响）
```

### C. 漂移累积追踪

跨章节追踪约束偏离的累积效应：

```
漂移追踪表（每章更新）：

| 章节 | 临时发明数 | 可疑 | 危险 | 约束偏离项 | 累积漂移度 |
|------|----------|------|------|----------|----------|
| ch1  | 2        | 1    | 0    | 无       | 0.05     |
| ch2  | 3        | 1    | 1    | 角色B新背景 | 0.15  |
| ch3  | 5        | 2    | 1    | 时间线偏移 | 0.30   |
| ...  |          |      |      |          |          |

累积漂移度 = Σ(每章可疑发明数×0.05 + 危险发明数×0.15)

阈值：
  0-0.20：正常范围 — 创作空间内的自然偏移
  0.20-0.40：警告 — 需要做一次约束对齐检查
  0.40-0.60：严重 — 必须暂停，重新对齐设计文档
  >0.60：失控 — 设计文档已与生成内容脱节，需要重做设计或重写
```

## 四、项目阶段评价（/retro 时使用）

### A. 项目总评（5 维度）

| 维度 | 评估内容 | 满分 |
|------|---------|------|
| 设计质量 | 设计就绪度 × 文档完整度 | 20 |
| 生成质量 | 各章 /check 平均分 | 20 |
| 约束健康 | 1 - 补丁率 - 连锁失效率 | 20 |
| 方法论遵守 | 流程偏差次数、步骤跳过次数 | 20 |
| 最终产出 | 整体故事的叙事力、主题表达、读者体验 | 20 |

```
项目总分 = 5个维度之和（满分 100）

评级：
  85-100：优秀 — 方法论充分发挥，产出质量高
  70-84：良好 — 方法论基本有效，有改善空间
  55-69：合格 — 方法论帮助有限，多处需改进
  <55：不合格 — 方法论存在结构性问题，需要进化
```

### B. 失败模式提取

从项目中系统性提取可学习的失败模式：

```
提取流程：

1. 收集本项目所有"危险发明"和"事后补丁"
2. 对每个失败点执行 5 Why 根因分析
3. 分类：
   - 方法论缺陷 → 进入 /evolve 候选
   - 执行缺陷 → 记录但不修改方法论
   - 能力缺陷 → 标记需要练习的领域
4. 对方法论缺陷：
   - 是否为新类型？→ 更新 failures/README.md 的分类法
   - 是否已有类似案例？→ 合并到已有 failure 条目
   - 是否涉及约束真空？→ 更新 constraint-contracts.md
5. 生成 RLHF 失败样本（如适用）
```

### C. 进化信号检测

从项目评价中自动提取方法论进化信号：

```
强信号（立即行动）：
  ✓ 项目总评 < 55 → 方法论有结构性问题
  ✓ 某维度连续 2 个项目得分最低 → 该维度的指导不足
  ✓ 约束健康维度 < 12 → 约束契约需要更新
  ✓ 同类失败在 failures/ 中出现 2+ 次 → 防护措施不充分

弱信号（观察中）：
  ~ 某文档质量评分低于其他文档 → 该文档可能需要重构
  ~ 跨文档一致性某对关系反复出分 → 接口约束可能缺失
  ~ 漂移累积度高于 0.30 → 设计文档的约束密度可能不够
  ~ 创造性偏离产出好结果 → 方法论可能过度约束

噪音（忽略）：
  ✗ 单个项目的单次偏差
  ✗ 文学质量维度的波动（主观性高）
  ✗ 新子类型首次尝试的低分（学习成本）
```

## 五、评价维度之间的关系

```
设计就绪度 ──────────────► 生成质量
    │                          │
    │   ┌──── 约束饱和率 ──┐   │
    │   │                  │   │
    ▼   ▼                  ▼   ▼
文档质量 ◄─── 跨文档一致性 ──► 约束遵守
    │                          │
    │     临时发明 ──► 漂移累积 │
    │                    │     │
    ▼                    ▼     ▼
项目总评 ◄────────── 补丁率/连锁率
    │
    ▼
进化信号 ──► /evolve
    │
    ▼
失败模式 ──► failures/
```

核心因果链：
```
文档质量高 → 约束密度充分 → 生成时临时发明少
                           → 约束遵守度高
                           → 补丁率低
                           → 连锁失效少
                           → 项目总评高
                           → 失败模式少
                           → 方法论不需要修改

文档质量低 → 约束密度不足 → 生成时大量临时发明
                           → 漂移累积快
                           → 需要事后补丁
                           → 补丁触发连锁失效
                           → 项目总评低
                           → 失败模式多
                           → 方法论需要进化
```

## 六、评价体系本身的进化

评价体系不是终态——它也需要进化。

```
每 3 个项目后检查：

□ 评分维度是否覆盖了所有重要的质量方面？
  → 如果某类问题反复出现但不在任何维度中 → 新增维度

□ 各维度的权重是否合理？
  → 如果高分项目仍有严重问题 → 相关维度权重偏低
  → 如果低分项目实际质量不错 → 相关维度权重偏高

□ 阈值是否校准？
  → 如果多数项目都能轻松通过阻塞阈值 → 阈值太低
  → 如果合理的项目被频繁阻塞 → 阈值太高

□ 跨文档一致性检查项是否完整？
  → 每次发现新的文档间矛盾 → 添加对应的检查项

□ 临时发明分类是否需要更新？
  → 新类型的临时发明被频繁标记为"可疑" → 需要更精确的分类
```
