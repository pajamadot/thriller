# 约束契约与饱和检查系统

> 方法论的验证粒度从"每个步骤对不对"下沉到"步骤之间的接缝有没有漏"
> 核心原则：故事的约束是一个网络，不是一条流水线

---

## 一、为什么需要约束契约

### 组合失效的本质

方法论中的每个步骤（`/trick`、`/characters`、`/structure`...）独立看都是完善的。
但故事的约束不是线性传递的——它们是网状的：

```
"为什么大家在一起"  需要  /trick (犯罪机会) + /characters (人物关系) + /setting (空间逻辑)
"犯罪是否可行"      需要  /characters (能力) + /setting (物理条件) + /trick (手法)
"嫌疑指数曲线"      需要  /characters (秘密) + /structure (信息释放) + /trick (线索布局)
"时间线一致性"      需要  /trick (犯罪时间) + /structure (叙事顺序) + /scene (场景内时间)
```

当一个约束跨越多个步骤但没有任何步骤明确"拥有"它时，就产生了**约束真空**。
约束真空在前期设计阶段不会被察觉，只会在文本生成后作为逻辑漏洞暴露。
此时补丁修复会触发连锁反应——因为约束之间存在隐性依赖。

### 事后补丁为什么必然失败

```
约束是一个网络，不是一个列表。

角色A的到场动机 → 依赖于 A和B的关系 → 依赖于 B的秘密 → 依赖于 B的到场动机
                                                              ↓
                                                    依赖于 聚合事件的性质
                                                              ↓
                                                    依赖于 犯罪的机会结构

修改任何一个节点 → 所有下游节点的一致性都需要重新验证。
在文本已生成的情况下，下游节点被具体文字锁死 → 拆东墙补西墙。
```

**解法**：在任何文本生成之前，通过约束契约和饱和检查确保所有约束都已被建立。

## 二、约束契约格式

每个步骤声明三份清单：

```
步骤 /X 的约束契约：

  保证输出 (Guarantees)：
    本步骤完成后，以下约束一定已被建立。
    ✓ ...

  前置依赖 (Requires)：
    本步骤正确执行的前提条件，必须由上游步骤提供。
    ← ...

  假设存在 (Assumes)：
    本步骤隐含依赖但不生产、也未明确要求上游提供的信息。
    ⚠ ...

"假设存在"是约束真空的预警信号：
  如果某个 ⚠ 项没有出现在任何步骤的 ✓ 清单中 → 约束真空 → 需要修复
```

## 三、当前步骤的约束契约

### `/start` 约束契约

```
保证输出：
  ✓ 子类型确定（决定后续步骤的权重和顺序）
  ✓ 篇幅和目标读者确定（约束后续所有步骤的尺度）
  ✓ 叙事视角确定（约束 /scene 的写作方式）
  ✓ 风格调性确定（约束全局语言风格）

前置依赖：
  ← 无（起始步骤）

假设存在：
  （无）
```

### `/theme` 约束契约

```
保证输出：
  ✓ 主题前提（一句话）
  ✓ 犯罪与主题的关系
  ✓ 侦探弧光与主题的关系
  ✓ 揭示与主题的关系

前置依赖：
  ← /start 的子类型（决定主题与子类型的适配方式）

假设存在：
  （无）
```

### `/trick` 约束契约

```
保证输出：
  ✓ 犯罪的完整真相（事实层）
  ✓ 诡计机制（表象如何形成）
  ✓ 线索布局（关键线索 + 误导线索的完整清单）
  ✓ 反派计划骨架（分步骤 + 痕迹）
  ✓ 公平性验证
  ✓ 知识状态矩阵框架

前置依赖：
  ← /theme 的主题前提（犯罪是主题的极端表现）

假设存在：
  ⚠ 犯罪发生的空间条件（封闭/开放、空间布局）
  ⚠ 犯罪发生的时间条件（为什么是这个时间点）
  ⚠ 人物为什么在犯罪现场（聚合前提）
  ⚠ 犯人与受害者之间有足够的接触机会
```

### `/premise` 约束契约（新步骤）

```
保证输出：
  ✓ 聚合事件的完整设计（事件、发起人、表面/真实目的）
  ✓ 每个角色的独立到场动机
  ✓ 到场动机的一致性验证（无循环依赖、无工具人）
  ✓ 空间封闭理由（如适用）
  ✓ 时间紧迫性（为什么是现在）
  ✓ "不来的代价"压力验证

前置依赖：
  ← /trick 的犯罪真相（知道谁是凶手才能设计"凶手为什么在"）
  ← /theme 的主题前提（聚合事件本身可以承载主题）

假设存在：
  ⚠ 角色的大致数量和身份雏形（/trick 中已有凶手/受害者设定）
```

### `/characters` 约束契约

```
保证输出：
  ✓ 侦探完整档案（方法论、缺陷、个人 Stakes）
  ✓ 犯人四维设计（动机、能力、伪装、弧光）
  ✓ 嫌疑人矩阵（动机、机会、能力、不在场证明）
  ✓ 受害者的 N 个面孔与秘密
  ✓ 关系网络图（两两关系 + 秘密链 + 冲突矩阵）
  ✓ 每个角色的动机罗盘（Lack/Yearning/Resistance/Desire）

前置依赖：
  ← /trick 的犯罪真相（凶手身份、犯罪方式决定角色设计）
  ← /premise 的聚合前提（到场动机约束角色的关系网络和机会结构）

假设存在：
  ⚠ 故事的时间跨度（影响角色弧光的设计空间）
  ⚠ 场景数量的大致范围（影响嫌疑指数曲线的细度）
```

### `/structure` 约束契约

```
保证输出：
  ✓ 三幕结构适配
  ✓ 章节大纲（每章核心事件 + 线索操作 + 嫌疑转移）
  ✓ 故事时间线 + 叙事时间线 + 两者的错位设计
  ✓ 悬念管理表

前置依赖：
  ← /characters 的完整角色系统
  ← /trick 的线索布局
  ← /premise 的聚合前提（第一幕的开场逻辑依赖于聚合事件）

假设存在：
  ⚠ 场景的物理空间是否足以支撑章节安排
  ⚠ 角色的活动范围是否与章节的地点设计一致
```

### `/scene` 约束契约

```
保证输出：
  ✓ 符合章节大纲的正文内容
  ✓ 线索的具体文本化呈现
  ✓ 知识状态的具体变化
  ✓ 嫌疑指向的具体变化

前置依赖：
  ← /structure 的章节大纲
  ← /characters 的角色声音和行为模式
  ← /trick 的知识状态矩阵
  ← /premise 的聚合前提（第一章必须自然呈现到场理由）

假设存在：
  ⚠ 无（/scene 是终端步骤，所有约束应已被上游建立）
```

## 四、约束饱和检查（Constraint Saturation Check）

### 何时执行

**在所有设计步骤完成、第一行正文生成之前**，执行一次完整的约束饱和检查。

### 检查方法

遍历所有步骤的"假设存在 ⚠"清单，验证每一条是否已被某个步骤的"保证输出 ✓"覆盖：

```
约束饱和矩阵：

| 被假设的约束                | 由谁假设      | 由谁保证     | 状态 |
|-----------------------------|---------------|-------------|------|
| 犯罪发生的空间条件          | /trick        | /premise ✓  | 饱和 |
| 人物为什么在犯罪现场        | /trick        | /premise ✓  | 饱和 |
| 角色的大致数量              | /premise      | /trick ✓    | 饱和 |
| 故事的时间跨度              | /characters   | ???         | 真空 |
| 场景的物理空间布局          | /structure    | ???         | 真空 |

状态分类：
  饱和：⚠ 项被某个 ✓ 项覆盖
  真空：⚠ 项没有对应的 ✓ 项 → 需要决定由谁负责
  冲突：⚠ 项被多个 ✓ 项覆盖且内容不一致 → 需要仲裁
```

### 处理约束真空

发现真空后的决策树：

```
发现约束真空 X
  ├── X 是否自然属于某个现有步骤？
  │   ├── 是 → 将 X 加入该步骤的"保证输出"清单，更新 SKILL.md
  │   └── 否 → X 是否跨越多个步骤？
  │       ├── 是 → 需要新增步骤或新增"跨步骤检查点"
  │       └── 否 → 将 X 加入最近的上游步骤
  │
  └── 记录到进化日志：
      - 约束真空描述
      - 发现方式（饱和检查 / 实践中暴露 / 盲区扫描）
      - 解决方案
      - 是否暴露了约束契约的系统性问题
```

## 五、跨步骤约束清单（可进化）

这份清单本身是一个可进化的文档。每次 `/retro` 发现新的"临时发明"，就往这里加一条。

### 当前已识别的跨步骤约束

```
1. 聚合前提（Gathering Premise）
   涉及步骤：/trick + /characters + /setting
   归属：/premise（新步骤）
   发现方式：密室类型生成后人物动机不足
   状态：已修复

2. 时间线一致性（Timeline Consistency）
   涉及步骤：/trick (犯罪时间) + /structure (叙事顺序) + /scene (场景内时间)
   归属：/structure（已在其输出中，但验证不够严格）
   发现方式：约束饱和检查
   状态：观察中

3. 空间可行性（Spatial Feasibility）
   涉及步骤：/trick (犯罪手法的空间需求) + /premise (聚合空间) + /structure (场景地点)
   归属：/premise 中的空间设计部分
   发现方式：约束饱和检查
   状态：已归属

4. 角色活动范围一致性
   涉及步骤：/characters (角色的行动能力) + /structure (章节的地点安排)
   归属：/structure 的章节大纲中应包含地点-角色交叉验证
   发现方式：约束饱和检查
   状态：待验证
```

## 六、约束契约的进化机制

### 契约本身如何更新

```
触发条件：
  1. /retro 的接口审计发现"临时发明" → 新增 ⚠ 或 ✓
  2. /blindspot 的组合盲区扫描发现新的跨步骤依赖 → 新增 ⚠
  3. /evolve 的结构级模式识别发现反复出现的接缝问题 → 调整契约归属
  4. 新增步骤时 → 创建新契约，更新所有相关步骤的 ← 和 ⚠

更新原则：
  - 每个 ⚠ 必须最终被转化为某个步骤的 ✓ 或被判定为不必要
  - ⚠ 项在清单中停留超过 3 个项目仍未归属 → 强制决策
  - 契约的变更必须记录在进化日志中
```

### 与质量门（Quality Gate）的集成

```
约束饱和检查应作为最高优先级的质量门：

QUALITY_GATES = [
  { name: 'constraint_saturation', severity: 'blocker' },  ← 新增：最高优先级
  { name: 'type_check', severity: 'blocker' },
  { name: 'graph_validation', severity: 'blocker' },
  { name: 'asset_integrity', severity: 'blocker' },
  { name: 'narrative_consistency', severity: 'warning' },
  ...
]

约束饱和检查不通过 → 不允许开始任何文本生成
```

## 七、接口审计执行指南

### 在 `/retro` 中使用

每次项目复盘时，回答以下问题：

```
接口审计问题清单：

1. 临时发明记录
   在创作过程中，哪些信息是"在某个步骤中临时编造的"，
   而不是从上游步骤的输出中继承的？
   | 临时发明的信息 | 发生在哪个步骤 | 应该由哪个步骤提供 |

2. 事后补丁记录
   创作过程中做了哪些事后修正？每个修正的根因是什么？
   | 补丁内容 | 触发原因 | 影响范围 | 根因归属（步骤/约束） |

3. 约束回溯分配
   对每个"临时发明"和"事后补丁"，它应该被归属到流程中的哪个步骤？
   → 如果无法归属 → 约束真空 → 需要修复契约或新增步骤

4. 连锁影响评估
   修补一个约束时，是否导致了其他约束的失效？
   → 如果是 → 这些约束之间存在隐性依赖 → 需要在契约中显式声明
```

### 在 `/blindspot` 中使用

将组合盲区检测融入定期盲区扫描：

```
组合盲区扫描步骤：

1. 契约完整性检查
   遍历所有步骤的约束契约，确认：
   □ 每个 ⚠ 都有对应的 ✓ 覆盖
   □ 没有新增的隐含假设未被记录

2. 跨步骤依赖发现
   对流程中每对步骤 (A, B)，问：
   "B 的正确执行是否依赖于 A 输出中未被显式列为 ✓ 的某些信息？"
   → 如果是 → 新增 ⚠ 到 B 的契约

3. 新故事类型压力测试
   选择一种从未使用过的子类型（如叙述诡计、倒叙推理），
   虚拟走一遍工作流，检查约束契约在该类型下是否仍然完备。
```

## 八、评价标准

### 约束系统健康度指标

```
饱和率 = 已覆盖的 ⚠ 项 / 全部 ⚠ 项
  目标：100%（发布前必须达到）

真空发现率 = 通过饱和检查发现的真空 / 通过实践暴露的真空
  目标：> 80%（大部分真空应在检查阶段发现，而非写作中暴露）

补丁率 = 事后补丁次数 / 约束总数
  目标：< 10%（低补丁率说明前置设计充分）

连锁失效率 = 触发连锁修改的补丁数 / 全部补丁数
  目标：< 20%（低连锁率说明约束之间的依赖已被显式管理）
```

### 进化判定标准

```
约束契约需要更新的信号（任一满足即触发）：
  ✓ 某个 ⚠ 项在 2+ 个项目中导致了临时发明
  ✓ 某个步骤的 ✓ 清单不完整（实际输出了契约未声明的约束）
  ✓ 新增子类型后饱和检查不通过
  ✓ 发现了契约中未记录的跨步骤依赖

约束契约不需要更新的信号：
  ✗ 只在 1 个项目中出现的问题（可能是执行问题）
  ✗ 可以通过更好的执行来解决的问题
  ✗ 约束真空已被现有步骤隐式覆盖且从未导致问题
```
