# 复盘方法论

> 从创作实践中提取可行动的方法论改进
> 参考 Agile 回顾会议方法论 + 反思性实践（Donald Schön）+ 刻意练习理论（Anders Ericsson）

---

## 一、复盘的三个层面

### 层面 1：发生了什么（事实层）

```
不做判断，只记录事实：

  时间线：
  - 各阶段实际花费的时间/精力
  - 卡点出现的位置和持续时间
  - 返工的位置和次数

  偏差记录：
  - 方法论建议的步骤 vs 实际执行的步骤
  - 跳过的步骤及原因
  - 额外增加的步骤及原因

  产出记录：
  - 各阶段的产出物及其质量自评
  - 最终作品的整体自评
```

### 层面 2：为什么发生（分析层）

```
对事实层的每个重要偏差/卡点进行根因分析：

  5 Why 法：
  问题：在第 12 章遇到了严重的线索矛盾
  Why 1：因为第 5 章埋设的线索 C03 与第 10 章的情节冲突
  Why 2：因为第 10 章的情节在写作中偏离了原始结构
  Why 3：因为结构设计阶段对第二幕下半的规划过于粗糙
  Why 4：因为方法论的结构模板在第二幕下半的指导密度不够
  Why 5：因为参考资料把重心放在了第一幕和中点，忽略了后半段
  → 根因：pacing-tension.md 对第二幕下半的指导不足

  区分：
  - 方法论层面的问题（规则有误/缺失/过于僵化）
  - 执行层面的问题（规则正确但没有遵守）
  - 能力层面的问题（方法论要求超出了当前创作能力）
  - 项目层面的问题（这个项目的特殊情况导致的）
```

### 层面 3：可以改善什么（行动层）

```
只有方法论层面的问题才进入改善清单。

每个改善建议必须包含：
  1. 具体修改点（改什么文件的什么内容）
  2. 修改方案（改成什么）
  3. 预期效果（改了之后会怎样）
  4. 验证方式（怎么知道改了之后更好了）
  5. 置信度（基于多少证据）
```

## 二、复盘维度清单

### 对每个维度进行评估

```
流程维度：
  □ 命令的顺序是否合理？是否需要调整？
  □ 是否有缺失的阶段（做了但方法论没有的步骤）？
  □ 是否有多余的阶段（方法论有但做了无用的步骤）？

参考资料维度：
  □ 每份参考资料被使用了几次？
  □ 使用时是否能快速找到需要的信息？
  □ 是否有需要参考但没有对应资料的时刻？
  □ 参考资料中的建议是否与实际写作冲突？

质量维度：
  □ /自检 的六个评分维度是否覆盖了所有重要方面？
  □ 评分标准是否清晰可操作？
  □ 分数最低的维度——是方法论指导不足，还是本次执行不佳？

创新维度：
  □ 这个项目中有哪些做法是方法论没有预见到的？
  □ 这些"意外做法"中有没有可以固化成方法论的？
  □ 有没有打破了方法论的规则却获得了更好效果的情况？
```

## 三、复盘会议结构（如果多人协作）

```
时间：项目完成后 1-3 天内（记忆还新鲜）
时长：45-90 分钟

流程：
  1. 开场（5分钟）：回顾项目概况
  2. 事实收集（15分钟）：每人写便签——发生了什么
  3. 情绪地图（10分钟）：标记过程中的高峰和低谷
  4. 根因分析（20分钟）：对关键卡点进行 5 Why
  5. 改善建议（15分钟）：生成方法论修改建议
  6. 优先级排序（10分钟）：选择 1-3 个最高优先级修改
  7. 行动项（5分钟）：谁来执行、什么时候完成
```

## 四、"前后对照"法

### 创作前预测 → 创作后比对

```
在 /开始 阶段记录预测：
  "我预计这个项目的最大挑战是___"
  "我预计方法论在___阶段最有帮助"
  "我预计需要偏离方法论的地方是___"

在 /复盘 阶段比对：
  预测准确 → 方法论的自我认知是准确的
  预测错误 → 方法论的自我认知需要校正
```

## 五、复盘反模式（常见陷阱）

```
1. "一切都很好"综合症
   问题：没有发现任何可改进的点
   原因：复盘太浅或者标准太低
   修复：强制要求至少找到 3 个改进点

2. "推翻重来"综合症
   问题：每次复盘都想重构整个方法论
   原因：把项目特殊问题当成了系统性问题
   修复：只有连续 3+ 个项目出现同样问题才允许结构性修改

3. "只看结果"综合症
   问题：最终作品不错就认为过程没问题
   原因：忽略了过程中的低效和痛苦
   修复：分别评估过程体验和产出质量

4. "归咎方法论"综合症
   问题：所有问题都归因于方法论不好
   原因：不区分方法论缺陷 vs 执行缺陷 vs 能力缺陷
   修复：使用三层归因模型，区分问题根源

5. "只改不验"综合症
   问题：改了方法论但从不验证修改是否有效
   原因：缺乏验证机制
   修复：每个修改必须有验证标准和验证时间点
```
